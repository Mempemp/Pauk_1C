# Чеклист перед коммитом (BSL)

## КОНТРОЛЬНЫЙ СПИСОК: Проверь перед коммитом

Используй этот список для проверки соответствия стандартам кодирования:

### Именование
- [ ] Все идентификаторы на русском языке в CamelCase
- [ ] Процедуры названы глагольными фразами (`ЗаполнитьТаблицу`)
- [ ] Функции названы существительными или вопросами (`ПолучитьДанные`, `ЭтоНовый`)
- [ ] Булевы переменные в утвердительной форме (`ЕстьОшибки`, `РазрешеноРедактирование`)
- [ ] Не затенён глобальный контекст (нет переменных `Документы`, `Справочники` и т.д.)

### Структура модуля
- [ ] Модуль разбит на области (`#Область ... #КонецОбласти`)
- [ ] Области идут в правильном порядке (ПрограммныйИнтерфейс → СлужебныеПроцедурыИФункции)
- [ ] Экспортные функции имеют комментарии с описанием параметров
- [ ] Области не вложены глубже 2 уровней

### Директивы компиляции
- [ ] Используется `&НаСервереБезКонтекста` вместо `&НаСервере` где возможно
- [ ] Чистые вычисления вынесены в `&НаКлиентеНаСервереБезКонтекста`
- [ ] Интерактивная логика (диалоги) только в `&НаКлиенте`

### Работа со строками
- [ ] Конкатенация в циклах заменена на массив + `СтрСоединить()`
- [ ] Используется `СтрШаблон()` для подстановки параметров
- [ ] Все строки для пользователя обёрнуты в `НСтр()`

### Комментарии
- [ ] Комментарии объясняют "зачем", а не "что"
- [ ] Экспортные функции имеют полное описание с параметрами и возвращаемым значением
- [ ] Нет устаревших комментариев (код изменён, комментарий остался старым)

### Безопасность и качество
- [ ] Нет использования `Выполнить()` или `Вычислить()`
- [ ] Магические числа вынесены в константы или перечисления
- [ ] Исключения не проглатываются (есть логирование и проброс)
- [ ] Одна процедура = одна ответственность (не более 50-100 строк)

### Клиент-сервер
- [ ] Бизнес-логика в модуле объекта, не в модуле формы
- [ ] Формы открываются через `ОткрытьФорму()`, не `ПолучитьФорму()`
- [ ] Серверные вызовы агрегированы (один вызов вместо нескольких)
- [ ] Используется `ТекущаяДатаСеанса()` вместо `ТекущаяДата()`

### Параметры проекта и архитектура (при доработке типовых и метаданных)
- [ ] Прочитан `.dev.env` (или параметры запрошены у человека) перед генерацией объектов и префиксов
- [ ] Маркеры доработки типового кода и префиксы согласованы с `reference/project-params-and-modifications.md`
- [ ] Нет лишней логики только в модуле формы; расширения и подписки использованы по смыслу (`reference/architecture-and-platform.md`)

### Производительность и граница клиент—сервер
- [ ] Нет цепочки подряд нескольких серверных вызовов с клиента, если достаточно одного возврата структуры/таблицы (`reference/runtime-performance-antipatterns.md`)
- [ ] Повторяющиеся в цикле тяжёлые обращения с теми же ключами закэшированы в `Соответствие` при необходимости
- [ ] Сопоставление двух таблиц по ключу — не вложенными циклами O(n²), а через индекс в `Соответствие`

### Ревью (по желанию команды)
- [ ] Замечания с низкой уверенностью не выдаются как жёсткие дефекты: для кода разумно требовать явные доказательства (воспроизведение, фрагмент плана, замер); сомнительные пункты — как вопросы, а не как блокер
