# Параметры проекта, оформление правок и именование метаданных

Индекс общих правил BSL — в `../SKILL.md`. Здесь — договорённости для **типовых** конфигураций и генерации метаданных при наличии файла параметров в корне проекта.

---

## 1. Файл параметров (`.dev.env`)

Перед задачей **с генерацией или массовым изменением метаданных и кода** прочитай `.dev.env` в корне прикладного репозитория. Если файла нет — **остановись** и запроси у человека префикс, организацию, шаблоны комментариев к доработке и прочие обязательные параметры. Подставлять значения «на глаз» запрещено.

Для задач **без изменения кода** (разбор, ревью, документация) отсутствие `.dev.env` само по себе не блокер, если не требуется генерировать новые объекты.

Типичные плейсхолдеры (имена полей согласовать с шаблоном в `.dev.env.example` проекта, если он есть):

| Параметр | Назначение |
|----------|------------|
| `{PREFIX}` | Префикс новых объектов метаданных, реквизитов, элементов форм, ролей |
| `{COMPANY}` | Шаблон комментария к доработке |
| `{DEVELOPER}` | Шаблон комментария к доработке |
| `{PLATFORM_VERSION}` | Минимальная/целевая версия платформы (влияет на доступность async/await, API) |
| `{COMMENT_OPEN}` / `{COMMENT_CLOSE}` | Открывающий и закрывающий маркеры доработки типового кода |
| `{NEW_OBJECTS_IN}` | Куда помещать новые объекты: основная конфигурация или расширение |

Номер задачи для комментариев (`{TASK}`) — из постановки; если не указан — уточнить у человека.

## 2. Стиль кода (дополнение к ИТС)

- **Отступы:** только табуляция, не пробелы.
- **Одна инструкция на строку.** Сложная логика в одной строке запрещена.
- **Выравнивание:** для группы присваиваний во **временные переменные** допускается выравнивание `=` пробелами; для цепочки присваиваний **свойствам объекта** через точку — не выравнивать, один пробел вокруг `=`.
- **Длина и сложность:** метод до ~200 строк (исключение — тексты больших запросов); глубина вложенности управляющих конструкций — меньше пяти уровней; число параметров функции — не более пяти (шестой и далее — через структуру-параметры).
- **Строки:** предпочитать `СтрШаблон()` / шаблонные строки вместо длинной конкатенации через `+` (простые `Префикс + Суффикс` допустимы, если так читабельнее).
- **Условия:** вложенные тернарные операторы `?()` запрещены; сложное условие из трёх и более частей — вынести в функцию.
- **Параметры функции** не использовать как скрытый «выход»; несколько значений — через возвращаемую структуру или явную структуру-параметр с конструктором `Новый Структура` / отдельная функция заполнения параметров.
- **TODO / FIXME:** ссылка на задачу, например `// TODO №14752: кратко суть`.
- **Псевдообласти комментариями** запрещены — только `#Область` / `#КонецОбласти`.

---

## 3. Комментарии к доработке типового кода

Маркеры применяются **только** при изменении типовых модулей типовой конфигурации.

- Открытие и закрытие — по шаблонам из `.dev.env`.
- После `//` обязателен пробел перед текстом маркера.
- Удалённый типовой код **не стирать** — закомментировать и рядом показать новую логику с префиксом `{PREFIX}` по правилам проекта.
- В **новых** процедурах внутри типового модуля маркер размещается внутри процедуры после заголовка.
- В модулях **полностью новых** объектов (имя с `{PREFIX}`) — не дублировать маркеры на каждую процедуру; достаточно **одного** блока в шапке модуля с описанием объекта доработки.

## 4. Именование метаданных

| Элемент | Правило |
|---------|---------|
| Новые объекты | Префикс `{PREFIX}` в имени |
| Синонимы | Без префикса; при конфликте — уточнение вроде суффикса организации по договорённости |
| Новые роли | С префиксом `{PREFIX}` |
| Подсистемы для доработок | Часто `{PREFIX}ДобавленныеОбъекты` / `{PREFIX}ИзмененныеОбъекты` — по стандарту проекта |
| Реквизиты и элементы **типовых** объектов | Префикс `{PREFIX}` |
| Внутри **нового** объекта (уже с префиксом в имени) | Реквизиты, ТЧ, элементы форм, команды, методы — **без** префикса |

Новые объекты помещать в подсистемы по стандарту проекта. Повторяющиеся составные типы — через определяемые типы.

### Выбор типа объекта (ориентир)

| Задача | Тип |
|--------|-----|
| Нормативно-справочная информация | Справочник |
| Хозяйственные операции | Документ |
| Количественные/суммовые остатки и обороты | Регистр накопления |
| Произвольные срезы по измерениям | Регистр сведений |
| Пользовательские отчёты | Отчёт (СКД) |
| Сервисные сценарии без постоянного хранения сущности | Обработка |
| Ограниченный перечень значений | Перечисление |

---

## 5. Описание экспортных процедур и функций

Для всех экспортных методов (кроме предопределённых обработчиков платформы) — блок комментария по стандарту ИТС: назначение, параметры (`Параметр - Тип - пояснение`), возвращаемое значение. Первая строка описания — с **глагола смысла** («Возвращает…», «Проверяет…»), не с «Процедура…» / «Функция…».

Для структур — перечислить ключи; для массивов — тип элемента.
