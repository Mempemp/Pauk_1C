# Архитектура, расширения и платформа

Индекс правил BSL по ИТС — в `../SKILL.md`. Здесь — **архитектурные** договорённости: куда класть логику, расширения, клиент-сервер, безопасность, запросы. Пересечение с `directives-forms-cs.md` и `error_handling` — не дублировать: при конфликте приоритет у узкого навыка (формы, ошибки).

---

## 1. Размещение кода

- Бизнес-логика — в **общих модулях** и модулях объектов, а не в модуле формы (исключения — только чисто представительская логика формы).
- Общий модуль сервера без состояния формы — по соглашению проекта суффиксы вроде `…ВызовСервера`, `…Сервер`, объектный/менеджерский контекст — по стандарту конфигурации.
- Клиентские общие модули — отдельно от серверных; форменные — по правилам проекта.
- Модуль объекта на сервере — обязательный препроцессор `#Если Сервер Или ТолстыйКлиентОбычноеПриложение Или ВнешнееСоединение Тогда` (или эквивалент по версии платформы).

### Приёмы структурирования

**Структура-результат:** несколько значений сразу — через `Новый Структура` и ключи (например `Выполнено`, `ТекстОшибки`).

**Ранний выход:** при отказе или пустых обязательных данных — `Возврат` / `Возврат ЗначениеПоУмолчанию` до основной логики, чтобы не раздувать вложенность.

**Поиск в таблице значений:** через `НайтиСтроки` с отбором-структурой вместо ручных циклов, где уместно.

**Подписки на события** предпочтительнее правок типовых модулей; обработчики — в общем модуле с префиксом проекта (см. `.dev.env`).

### Нормализация входа

На границе API приводить массив/одиночную ссылку к единому виду (массив), проверять тип при критичности. Для структур — доступ к ключу через `Свойство()` или проверенное получение, чтобы не падать на отсутствующем ключе.

## 2. Расширения и типовой код

Порядок предпочтения вмешательства:

1. Подписки на события.
2. Расширение конфигурации.
3. Прямая правка типового модуля — только если иначе нельзя и с маркерами доработки по `project-params-and-modifications.md`.

В расширениях директивы `&Перед` / `&После` предпочтительнее `&Вместо`; для `&Вместо` — только для функций и с обязательным `ПродолжитьВызов()` где требует платформа.

Типовые роли **не изменять** — добавлять новые с префиксом проекта.

Куда помещать новые объекты (`{NEW_OBJECTS_IN}` из `.dev.env`): основная конфигурация или расширение — соблюдать значение параметра; по умолчанию в типовых договорённостях часто основная конфигурация, расширение — для перехвата событий.

Визуальное редактирование форм в расширении минимизировать; изменения форм по возможности программно.

## 3. Платформа и клиент-сервер

### Асинхронность и модальность

Модальные вызовы, блокирующие интерфейс там, где для управляемого приложения предусмотрена асинхронная модель, избегать. Конкретный API (оповещения, `Ждать`, асинхронные методы) — в рамках `{PLATFORM_VERSION}` из `.dev.env`: не смешивать в одной процедуре асинхронный контракт с синхронными блокирующими вызовами.

Диалоги на сервере не выполнять.

### Вызовы клиент ↔ сервер

Для серверных методов **без** чтения/записи данных формы использовать `&НаСервереБезКонтекста` (или эквивалент по версии). Если нужен только реквизит формы — передать значение параметром и остаться без контекста формы, где возможно.

### Безопасность

`Выполнить` / `Вычислить` — не использовать без крайней необходимости и явного обоснования в задаче.

Учётные данные, токены, ключи — не хранить в коде (см. красные линии в `A-INVARIANTS.md`). RLS и права закладывать в дизайн.

### Ошибки и строки

Пользовательские строки — через `НСтр()` и подстановки по стандарту проекта. Пустые обработчики `Исключение` запрещены: логирование и/или проброс. Для текста ошибки в журнал предпочитать развёрнутое описание по правилам проекта и платформы.

### Даты на сервере

Использовать `ТекущаяДатаСеанса()` (или актуальный аналог по версии), а не «текущую дату» без учёта часового пояса сеанса — согласовано с чеклистом навыка.

### Запросы (архитектурный уровень)

Временные таблицы в тексте запроса — с префиксом имени вроде `ВТ_`. По возможности `ВНУТРЕННЕЕ СОЕДИНЕНИЕ` вместо `ЛЕВОЕ`, если внешняя сторона не нужна. Движения регистров не править напрямую — только через проведение документов и предусмотренные механизмы.

### Кроссплатформенность

COM-объекты и привязка к конкретному Office на сервере — только если это явно требование задачи. Пути — через платформенные функции и разделители, без жёсткой привязки к `\` в строках.

Перед использованием нового метода платформы убедиться, что он доступен в `{PLATFORM_VERSION}` из `.dev.env`.

## 4. Запахи кода (сигналы к рефакторингу)

| Запах | Признак | Направление исправления |
|-------|---------|-------------------------|
| Сгусток данных | Одинаковые 3+ параметра ходят вместе по методам | Структура-параметр, конструктор параметров |
| Примитивизация | Коды вместо ссылок, «магические» строки вместо перечислений | Перечисления, ссылки, определяемые типы |
| Разъезжающееся изменение | Один модуль правят по несвязным причинам | Разделение ответственности (SRP) |
| Дробострельность | Одна бизнес-правка требует правок в множестве несвязных местах | Свести логику в один/несколько связных модулей |
| Зависть к данным | Метод формы в основном крутит чужой объект | Перенос в модуль объекта / общий модуль объекта |
| Повторное использование переменной | В одной переменной по ходу метода разные по смыслу значения | Отдельные переменные по смыслу |

## 5. Архитектурные антипаттерны (уровень подсистемы)

| Антипаттерн | Признаки | Риск |
|-------------|----------|------|
| «Комок грязи» | Нет явных границ подсистем, всё зависит от всего | Высокая стоимость изменений, непредсказуемые регрессии |
| «Бог-модуль» | Один модуль на сотни несвязных процедур, разные причины правок | Сложно тестировать, смерживать и разбирать ответственность |
| Жёсткая связность | Прямые зависимости от чужих деталей реализации, обход слоёв | Локальная правка вызывает лавину правок |
| Копипаст-архитектура | Один и тот же код в множестве мест без общих модулей | Расхождение поведения, рост стоимости сопровождения |
| Преждевременная оптимизация | Сложные кэши и абстракции без замера и без явной потребности | Лишняя сложность без выигрыша |

Порядок вмешательства (подписки → расширение → правка типового) — в разделе 2 выше.
