# Выгрузка метаданных и объект конфигурации

Индекс навыка — [../SKILL.md](../SKILL.md). Инструменты MCP (имена, порядок) — [../../../mcp/README.md](../../../mcp/README.md).

## 1. Структура выгрузки

Корень выгрузки обычно содержит **`Configuration.xml`** и каталоги по типам: `Catalogs/`, `Documents/`, `InformationRegisters/`, `Enums/`, `CommonModules/` и т.д.

У объекта путь вида `Catalogs/ИмяОбъекта/ИмяОбъекта.xml` плюс при необходимости подкаталоги `Forms/`, `Ext/`, модули `.bsl`.

Любая правка должна сохранять **согласованность** с `Configuration.xml` (регистрация в `ChildObjects`) и **ссылочную целостность** типов (`CatalogRef.…`, `DocumentRef.…` и т.д.).

## 2. Типы объектов (для ориентира)

Типичный набор, с которым работают сценарии «метаданные + валидация»:

- Справочники, документы, перечисления, планы обмена, планы счетов, планы видов характеристик, планы видов расчёта, бизнес-процессы, задачи.
- Регистры сведений, накопления, бухгалтерии, расчёта.
- Отчёты, обработки, общие модули, константы, определяемые типы, подписки на события, журналы документов.
- HTTP- и веб-сервисы (вложенная структура в XML).

Точное перечисление полей и свойств — в платформенной документации и в **XSD** для типа (через MCP `get_xsd_schema`, если доступен).

## 3. JSON-DSL (черновик генерации)

Часть команд автоматизации использует **JSON-описание** корня:

```json
{
  "type": "Catalog",
  "name": "Валюты",
  "synonym": "Валюты",
  "attributes": ["Код: String(3) | req", "Наименование: String(150)"]
}
```

Сокращения атрибутов: `"Имя"`, `"Имя: Тип"`, `"Имя: Тип | req, index"` и русские синонимы типов (`Строка`, `Число`, `СправочникСсылка.X` и т.д.) — как в типовых генераторах; детали DSL привязывайте к **вашей** утилите или копируйте из актуальной спецификации инструмента, если он есть в репозитории.

После генерации или ручной правки XML **обязательна проверка** (ниже).

## 4. Валидация объекта метаданных

### 4.1 Через MCP (предпочтительно)

1. **`get_xsd_schema`** для нужного типа объекта (или `Форма` для форм).
2. **`verify_xml`** для строки XML или фрагмента по контракту инструмента.

### 4.2 Логические проверки (ручной чеклист)

Имеет смысл пройтись по контролю качества структуры:

| # | Проверка |
|---|-----------|
| 1 | Корневой элемент XML и версия формата, well-formed |
| 2 | `InternalInfo` / тип генерируемого модуля при необходимости |
| 3 | Имя, синоним, обязательные свойства типа |
| 4 | Допустимые значения перечислимых свойств платформы |
| 5 | Стандартные реквизиты, если задействованы |
| 6 | Дочерние узлы: допустимые типы дочерних объектов |
| 7 | Реквизиты, измерения, ресурсы: UUID, имя, тип |
| 8 | Уникальность имён в пределах объекта |
| 9 | Структура табличных частей |
| 10 | Согласованность взаимозависимых свойств |
| 11 | Вложенность для HTTP/Web-сервисов |

Перед **удалением** объекта из выгрузки — поиск ссылок через MCP (`find_usages_of_object`, `trace_impact`, `search_code` и т.д. по `pauk/mcp/README.md`).

## 5. Загрузка в конфигурацию

Обратная загрузка изменённой выгрузки в ИБ выполняется **Конфигуратором**, **EDT** или **скриптами сборки вашего проекта**. Этот навык **не** задаёт путей к платформе и не заменяет регламент команды.

## 6. Утилиты командной строки

Скрипты вида **`1c-meta-compile`**, **`1c-meta-validate`**, **`1c-meta-edit`** из внешних наборов правил **не входят** в копируемый пакет Pauk. Если команда их подключает — описание запуска и параметров храните **в своём репозитории** (CI, `README`, `mcp.json` не трогаем здесь).
