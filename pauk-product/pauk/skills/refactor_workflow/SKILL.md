# Рефакторинг: гибридный порядок и инструменты

## Когда применять

- **Перестройка** существующей логики: вынесение в общие модули, **переименование** процедур/сущностей, смена сигнатур, **узкая** смена контракта.
- В задаче явно: **рефакторинг**, **вынести в общий модуль**, **разорвать связность**, **переименовать** объект или метод, **разбить** монолит.
- Для **полной новой фичи** с нуля сначала политика готовности и `scale` по **`PROTOCOL.md`**; этот навык — про **сохранение** поведения при перестановке кода.

## Методология: сверху вниз, затем снизу вверх

1. **Top-down:** понять **бизнес-смысл** и границы: явно проследить **цепочки вызовов** и **движение данных** (от пользовательского сценария до нижнего уровня).
2. **Bottom-up:** **рефакторить** с **самого низкого** уровня — вспомогательные функции, точечные API без «протекания» наверх лишних зависимостей.
3. **Сборка:** подключить обновлённые части **в** высокоуровневые процедуры, прогнать **критерии** из политики проекта ([A-INVARIANTS.md](../../policy/A-INVARIANTS.md)) и **регрессию** в рамках задачи.

## Инструменты: без дублирования длинного справочника

**Порядок:** граф/метаданные MCP → [README MCP](../../mcp/README.md) (разделы **Refactoring** и **Tool priority: graph-metadata → code-metadata → Grep**) → **`grep`/`read_file` только** если инструменты выгрузки/графа недоступны или **исчерпаны**.

Полные имена (`get_object_dossier`, `trace_impact`, `search_code` ↔ `codesearch` и т.д.), параметры глубины и **контроль** после рефакторинга — **только** в **`pauk/mcp/README.md`**, с учётом **фактически** подключённых серверов.
