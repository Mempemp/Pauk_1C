# Интеграции 1С с внешними системами

## Когда применять

- Обмен с **внешним API**, очередями, файлами, сервисом, **не** реализуемый целиком средствами платформы в одной сессии мыслительно.
- Нужен **двухконтурный** путь: проверить протокол, форматы, ошибки, **потом** встроить в конфигурацию.
- В задаче явно: **интеграция**, **HTTP/REST**, **обмен**, **клиент вне 1С**, **прототип** обмена.

## Критический принцип

**Сначала** отладить **контракт и бизнес-логику** на стороне, где **быстрее цикл** правок и тестов (как правило **Python**: запросы, разбор JSON/XML, валидация, повторяемые прогоны). **Затем** перенести проверенные шаги в **BSL**, не дублируя ошибок прототипа.

## Как развивать сценарий

1. Зафиксировать границы: формат тел, коды ответа, **идемпотентность**, лимиты, **секреты** только из безопасного хранилища проекта, не в тексте чата.
2. **Прототип** (Python или другой выбранный в проекте стек) — интеграционные сценарии, негативные кейсы, **повтор** под нагрузкой с учётом регламента команды.
3. **BSL**: тонкий слой взаимодействия с платформой (сессия, объекты метаданных) и вызовов, совпадающий по шагам с **уже отлаженным** прототипом.
4. Нюансы **таймаутов, повторов, транзакций, журналирования** внешних сбоев — в навыке [error_handling/SKILL.md](../error_handling/SKILL.md) вместе с [coding-standards/SKILL.md](../coding-standards/SKILL.md).

## С чем сочетать

| Тема | Куда |
|------|------|
| Стиль BSL, директивы, `Попытка` | `coding-standards` |
| HTTP в транзакции, блокировки, ЖР | `error_handling` |
| Поиск по коду, регистрации движений, если задействованы граф/метаданные | [README MCP](../../mcp/README.md) (имена и порядок — по локальной схеме) |

**Не** подменять этим навыком: запросы к **СУБД** (навык `query-patterns`); **БСП**-типовые пакеты (`ssl_patterns`); **только** модуль формы (`form_patterns`).
