Индекс навыка — [../SKILL.md](../SKILL.md). Валидация XML формы и граница с модулем — [../../metadata_manage/reference/form-xml-validate.md](../../metadata_manage/reference/form-xml-validate.md). Конвейер «аналог → схема → правка → проверка» — [form-metadata-xml-pipeline.md](form-metadata-xml-pipeline.md).

## Когда править именно Form.xml как разметку

Имеет смысл трогать **`Form.xml` как структуру**, когда задача **не** в коде `Form.Module.bsl`, а в:

- **элементах управления** (иерархия, группы, таблицы, поля ввода, декорации);
- **реквизитах формы** (объявление, типы, связь с данными через пути привязки);
- **командах** формы (объявление, привязка к кнопкам панелей и т.п.).

Логика обработчиков, директивы, вызовы сервера — в **модуле формы**; после правок XML нужно **согласовать** имена процедур с модулем ([form-xml-events.md](form-xml-events.md)).

## MDClasses (метаданные) и logform (`Form.xml`)

Принцип тот же, что для корневого XML объекта и формы внешней обработки — [epf_erf_external/reference/root-metadata-and-forms.md](../../epf_erf_external/reference/root-metadata-and-forms.md):

| Файл | Схема / модель | Смысл |
|------|----------------|--------|
| **`…/ИмяОбъекта.xml`** справочника, документа, **`Configuration.xml`** и т.п. | **Классы метаданных** (**MDClasses**), корень **`MetaDataObject`** | Реквизиты объекта, `ChildObjects`, ссылки на формы — **не** разметка полей на форме. |
| **`Form.xml`** | **Логическая форма** (**logform**): своё пространство имён и XSD | Элементы управления, реквизиты **формы**, команды, привязки данных — **другая** схема; узлы из MD в logform **не копировать** «как есть». |

Где лежит `Form.xml`, зависит от **носителя**: `…/Forms/…/Ext/Form.xml` у объекта, отдельный каталог у общей формы; запись в `ChildObjects` корня или объекта должна совпадать с фактическим деревом.

При автогенерации из JSON/DSL **не смешивайте** MD и logform и **не путайте** шаблоны путей (форма отчёта vs форма справочника) — ориентир: **аналог той же роли** в дереве выгрузки проекта.

## Генерация из JSON/DSL и проверка по XSD

Если форма или её фрагмент собираются **из JSON, DSL или шаблона**:

1. Зафиксируйте **версию схемы** формы в XML (атрибут/узел версии в корне `<Form>`) в соответствии с **платформой проекта**.
2. Получите **XSD** для типа «форма» через **MCP** (инструменты вроде `get_xsd_schema` — только как описано в [README MCP](../../../mcp/README.md) для подключённого сервера).
3. Прогоните **`verify_xml`** (или эквивалент по контракту) по сгенерированному `Form.xml`.
4. Исправьте расхождения до успешной проверки; затем загрузку в ИБ выполняйте по процессу проекта.

Целостность объекта-носителя (не только формы) при массовых правках — **[metadata_manage/SKILL.md](../../metadata_manage/SKILL.md)**; обзор **корня** `Configuration.xml` и `ChildObjects` — **[configuration_structure/SKILL.md](../../configuration_structure/SKILL.md)**.
