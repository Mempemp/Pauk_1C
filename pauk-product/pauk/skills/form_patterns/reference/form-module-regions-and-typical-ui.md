## 1. Области модуля управляемой формы

Все перечисленные области **обязательны** в модуле формы, даже если внутри пока нет кода. Если на форме **несколько табличных частей** — для **каждой** заводится своя область обработчиков с **уникальным** именем по имени таблицы.

**Внутри процедур и функций** псевдообласти комментариями не использовать — только `#Область` / `#КонецОбласти` на уровне модуля.

### Рекомендуемый каркас (имена областей — по смыслу ИТС)

```bsl
#Область ОбработчикиСобытийФормы

#КонецОбласти

#Область ОбработчикиСобытийЭлементовШапкиФормы

#КонецОбласти

#Область ОбработчикиСобытийЭлементовТабличнойЧастиТовары

#КонецОбласти

#Область ОбработчикиКомандФормы

#КонецОбласти

#Область СлужебныеПроцедурыИФункции

#КонецОбласти
```

Имя области для ТЧ замените на фактическое имя таблицы формы (`...ТабличнойЧастиИмяТаблицы`).

## 2. Типовые формы: программно, а не «визуально»

Доработка типовой формы в конфигурации по устойчивой практике выполняется **программно** (при создании на сервере, через подписку на событие или расширение по правилам проекта), а не только перетаскиванием элементов в конфигураторе без кода.

### Размещение добавленных элементов

- Если на форме есть **страницы** — вынести доработку на отдельную страницу (например «Дополнительно» или с префиксом из параметров проекта).
- Если вкладок нет — сгруппировать добавленные поля в **группу без заголовка**.
- Имена добавленных элементов на **типовой** форме — с **префиксом доработки** проекта (см. параметры в `.dev.env`).

### Новые формы (нетиповые объекты)

- Разнести шапку и табличные части по **отдельным страницам**: основная — шапка, далее — по одной странице на каждую ТЧ при необходимости.
- Для страниц с ТЧ задать **путь к данным заголовка** (свойства страницы в метаданных формы).
- Поля ссылочного типа — разумное ограничение ширины (ориентир до **27** символов в подписи/поле по соглашению проекта).
- Многострочные комментарии — задать высоту и ширину осознанно (например ширина около **79**, высота **3** — как стартовый ориентир, подстройте под ИТС проекта).

### Проверка заполнения

- Использовать свойства проверки заполнения у реквизитов формы, где это уместно.
- Перед записью при необходимости вызывать **`ПроверитьЗаполнение()`** на клиенте/сервере по сценарию платформы:

```bsl
Если Не ПроверитьЗаполнение() Тогда
    Возврат;
КонецЕсли;
```

### Команды формы

- Для команд, которые **изменяют сохраняемые данные**, включать признак **«Изменяет сохраняемые данные»** в метаданных команды.
