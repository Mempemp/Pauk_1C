# Чеклист: транзакции, ошибки, блокировки

Вместе с [checklist-precommit.md](../coding-standards/checklist-precommit.md). Правила **запросов** (в т.ч. соединения и агрегаты) — в отдельном навыке по запросам.

## Транзакции

- [ ] `НачатьТранзакцию()` стоит **сразу** перед `Попытка`
- [ ] `ЗафиксироватьТранзакцию()` — **последняя** успешная операция в `Попытка`
- [ ] `ОтменитьТранзакцию()` — **первая** операция в `Исключение` (если нет согласованной вложенной схемы с отдельным откатом)
- [ ] Запись в журнал регистрации по ошибке — **после** `ОтменитьТранзакцию()`
- [ ] Нет кода между `ЗафиксироватьТранзакцию()` и концом `Попытка`

## Исключения

- [ ] В `Исключение` есть осмысленная обработка: журнал и/или проброс, а не пустой блок
- [ ] В журнал — `ПодробноеПредставлениеОшибки(ИнформацияОбОшибке())` (или эквивалент по стандарту проекта)
- [ ] Пользователю — краткий текст; не оставлять только сырой стек на границе UI
- [ ] Явный выбор: `ВызватьИсключение;` vs `ВызватьИсключение Текст;` по правилам из [exceptions-journal-user.md](reference/exceptions-journal-user.md)

## Блокировки

- [ ] `БлокировкаДанных` установлена **до** чтения данных, которые затем меняются по результату чтения
- [ ] При пакетной блокировке нескольких объектов — **один и тот же порядок** (например сортировка ссылок)
- [ ] В транзакции нет долгих операций без необходимости (запросы на весь периметр, HTTP, ожидание пользователя)

## Пакетная обработка

- [ ] Ошибка по одному элементу не рвёт цикл без явной на то политики
- [ ] Каждая ошибка пакета попала в журнал (или в накопитель с последующим отчётом)
- [ ] Пользователю понятен итог: сколько обработано, сколько с ошибками
