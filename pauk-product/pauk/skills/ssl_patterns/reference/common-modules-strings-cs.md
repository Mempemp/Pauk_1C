# Правила 1–3: ОбщегоНазначения, строки, ОбщегоНазначенияКлиентСервер

## Правило 1: модуль `ОбщегоНазначения` — начинать с него

`ОбщегоНазначения` — типичный **центральный** набор вспомогательных вызовов. Перед написанием **своей** функции проверь, не покрыт ли сценарий **уже** библиотекой. Дублирование смысла с БСП — антипаттерн.

| Функция | Назначение |
|---------|------------|
| `ЗначениеРеквизитаОбъекта()` | один реквизит по ссылке — вместо обращения через цепочку точек «наугад» |
| `ЗначенияРеквизитовОбъекта()` | **несколько** реквизитов **одной** ссылки — **одним** запросом, не серией чтений |
| `ЗначенияРеквизитовОбъектов()` | **один** реквизит (или набор полей) сразу для **массива ссылок** — пакетное чтение вместо цикла с точками по каждой ссылке |
| `СообщитьПользователю()` | сообщение с привязкой к полю — предпочтительнее сырого `Сообщить()` для пользовательских сценариев |
| `МенеджерОбъектаПоСсылке()` | менеджер по ссылке |
| `ПодсистемаСуществует()` | условное подключение подсистемы |
| `ОбщийМодуль()` | динамическое получение ссылки на модуль |
| `ЭтоСсылка()` / `СсылкаСуществует()` | проверки ссылок |

### Точечная нотация по ссылкам (часто хуже)

Несколько обращений `Ссылка.Реквизит1`, `Ссылка.Реквизит2` могут дать **лишние** чтения. **Пакет** через `ЗначенияРеквизитовОбъекта()` (или `ЗначениеРеквизитаОбъекта` для одного) — **практичнее**, если это соответствует **вашим** соглашениям в конфигурации.

**Пример — пакет реквизитов:**

```bsl
// нежелательно: много чтений через точки (зависит от контекста, часто N обращений)
// предпочтительно: один пакетный вызов БСП, где применимо
РеквизитыКонтрагента = ОбщегоНазначения.ЗначенияРеквизитовОбъекта(
    КонтрагентСсылка,
    "Наименование, ИНН, ОсновнойМенеджер");
Наименование = РеквизитыКонтрагента.Наименование;
ИНН = РеквизитыКонтрагента.ИНН;
Ответственный = РеквизитыКонтрагента.ОсновнойМенеджер;
```

## Правило 2: `СтроковыеФункции*` — строки

Используй `СтроковыеФункцииКлиентСервер` (или вариант на **сервере** по правилам вашей поставки) для подстановок, склонений, проверок входных данных — там **учтены** типовые пограничные случаи.

| Функция | Смысл |
|---------|--------|
| `ПодставитьПараметрыВСтроку()` | подстановка в шаблон (аналоги «шаблона с плейсхолдерами») |
| `СтрокаСЧисломПредметов()` | согласование с числом: «1 документ», «2 документа» |
| `ЕстьНедопустимыеСимволы()` | валидация ввода |
| `ТолькоЦифрыВСтроке()` | ИНН, номера и т.д. |
| `РазложитьСтрокуВМассивПодстрок()` | разбор по разделителю |

**Пример:**

```bsl
ТекстСообщения = СтроковыеФункцииКлиентСервер.ПодставитьПараметрыВСтроку(
    НСтр("ru = 'Обработано %1 из %2 документов. Ошибок: %3.'"),
    Формат(Обработано, "ЧГ="),
    Формат(Всего, "ЧГ="),
    Формат(КоличествоОшибок, "ЧГ="));

ТекстОповещения = СтроковыеФункцииКлиентСервер.СтрокаСЧисломПредметов(
    КоличествоДокументов,
    НСтр("ru = 'документ, документа, документов'"));
```

## Правило 3: `ОбщегоНазначенияКлиентСервер` — общий код

Модуль с `&НаКлиентеНаСервереБезКонтекста` даёт **один** и тот же код **и** в клиенте, **и** в серверных общих.

| Функция | Смысл |
|---------|--------|
| `СвойствоСтруктуры()` | чтение ключа с **значением по умолчанию** |
| `ДополнитьМассив` / `ДополнитьСтруктуру` / `ДополнитьТаблицуИзМассива` | агрегация коллекций |
| `ПроверитьПараметр()` | проверка с понятной ошибкой |

**Пример — безопасно из структуры:**

```bsl
ДатаНачала = ОбщегоНазначенияКлиентСервер.СвойствоСтруктуры(
    ПараметрыОтчёта,
    "ДатаНачала",
    НачалоГода(ТекущаяДатаСеанса()));
```

Прямой доступ `ПараметрыОтчёта.Поле` **без** гарантии, что поле в структуре есть, даёт риск ошибки.
