# Правило 4: как искать готовую функциональность БСП

**Смысл:** в разных поставках БСП **много** кода; без порядка легко дублировать то, что уже сделано в библиотеке. Порядок **не** привязан к конкретной IDE: сначала **репозиторий/выгрузка** с **той** версией конфигурации, в котором ведётся работа, затем — **средства семантического/символьного поиска по коду** (если подключены), **документация 1С (ИТС)** — когда нужно **уточнить** семантику, ограничения и поведение уже найденной конструкции.

## 1. Поиск в проекте (приоритет)

- Искать в **тексте** исходников: общие модули, объявления `Функция`, известные имена подсистем БСП.
- Ориентир по структуре: ветка репозитория с **той** поставкой, которую сопровождаешь; типично `CommonModules/` и связанные каталоги.
- **Цель:** убедиться, что **в этой** ревизии уже есть готовая процедура, прежде чем писать свою.

**Практическое ограничение:** в больших выгрузках **один** только `grep` по всему дереву даёт усталость и шум. Имеет смысл **сужать** область (модуль, подсистема) и держать в уме **версию** поставки — поведение БСП **меняется** от релиза к релизу.

## 2. Средства поиска по коду (MCP, при наличии)

Если в рабочей среде подключены **MCP** или плагины уровня **символьного / семантического** поиска по репозиторию (в т.ч. **Serena**, **RLM-tools** и аналоги — **конкретные имена** зависят от стенда) — **после** сужения зоны в проекте используй их для **быстрого** уточнения, где реализованы вызовы, похожие на нужный сценарий. Это **не** заменяет проверку на **твоей** поставке, а ускоряет навигацию.

**Поиск в платформенном** MCP 1С (`search` / `info`) полезен для **методов и типов платформы**; **имена** и состав модулей **БСП** зависят от конфигурации — опирайся в первую очередь на **фактический** код **этой** базы/выгрузки.

## 3. ИТС и «последняя очередь»

- Документация 1С — когда нужно **понять смысл**, нюансы, ограничения или типовой сценарий для уже **найденного** API.
- **Не** начинать с длинного чтения ИТС **вместо** просмотра **своей** поставки: риск рассинхрона **версий** (описание одной ветки, в проекте — другая).

## 4. Когда писать своё

| Ситуация | Решение |
|----------|---------|
| В **этой** выгрузке БСП есть подходящая функция | **Брать БСП** |
| Есть похожая, с «лишним» | Обычно **всё равно БСП** — единообразие важнее |
| Нет подходящего в **вашей** поставке | **Своя** реализация в **стиле** БСП; при обновлении поставки пересмотр |
| Конфигурация **без** БСП | Правила этого навыка **не** про «типовую библиотеку» — своё решение по месту |

## 5. Долгий горизонт (организационно)

Если из-за **объёма** кода и **леса версий** БСП навигация **регулярно** тормозит, **на уровне команды** уместны снимок/индекс кода **той** ревизии, с которой ведёте сопровождение, и политика **обновления** этого снимка при смене релиза. Конкретный механизм (MCP, отдельная база, иное) здесь **не** фиксируется — решает команда по своему контексту.
