# Концептуальный цикл: каркас → наполнение → epf/erf

Работа с внешними обработками и отчётами в виде каталога XML обычно следует одному и тому же **концептуальному циклу**, без привязки к конкретным именам утилит или скриптов из других репозиториев — те задаются **в вашем проекте** (репозитории прикладной разработки, CI, локальные обёртки над Конфигуратором или EDT).

## 1. Каркас

Получить согласованное дерево:

- корневой **`MetaDataObject`** с нужным типом (**`ExternalDataProcessor`** или **`ExternalReport`**);
- минимальный **`ChildObjects`** и **`Properties`**, включая форму по умолчанию, если форма уже создана;
- **`Ext/ObjectModule.bsl`** при необходимости — даже пустой модуль с корректной структурой области.

На этом шаге критичны **UUID**, **версия формата** и отсутствие «висячих» ссылок из корня на несуществующие дочерние имена.

## 2. Добавить формы и макеты

- Создать каталоги **`Forms/<Имя>/Ext/`** с **`Form.xml`** и модулем формы по правилам управляемой формы.
- Добавить **`Templates/<Имя>/Template.xml`** для табличных макетов; синхронизировать имена с **`ChildObjects`** и ссылками из формы или СКД.
- Для **внешнего отчёта** — подключить контур **СКД** и при необходимости **варианты**: все новые узлы должны быть отражены в метаданных отчёта, иначе загрузка или выполнение отчёта разойдётся с файлами на диске.

## 3. Сборка epf / erf

Преобразование каталога XML во **внешний файл** выполняется **инструментами платформы** (загрузка из файлов XML в Конфигураторе, сценарии EDT, или **ваши** скрипты, которые вызывают те же операции). Обратная операция — **выгрузка** бинарного epf/erf обратно в XML для правки в системе контроля версий.

Рекомендуется фиксировать в проекте **один утверждённый способ** сборки и проверки (чтобы версия платформы, ключи загрузки и порядок каталогов совпадали между разработчиками), но сами имена исполняемых файлов и параметры командной строки в общий пакет навыков **не входят** — см. документацию вашего репозитория.

## 4. Проверка после изменений

После правок XML имеет смысл прогонять **валидацию XML** там, где это принято в проекте (XSD, встроенные проверки, загрузка в чистый контур), и открыть внешний файл в целевой версии платформы — особенно если менялись **`ChildObjects`**, СКД или версия формата.
