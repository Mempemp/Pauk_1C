# Корень выгрузки: Configuration.xml и окружение

Индекс навыка — [../SKILL.md](../SKILL.md). Объекты в каталогах и согласованность с корнем — [../../metadata_manage/reference/meta-export-overview.md](../../metadata_manage/reference/meta-export-overview.md).

## 1. Структура выгрузки (логический состав)

Типичный корень каталога выгрузки в файлы содержит:

| Компонент | Назначение |
|-----------|------------|
| **`Configuration.xml`** | Описание **всей конфигурации** как одного узла метаданных: свойства, дочерние объекты по типам, служебные блоки. |
| **`ConfigDumpInfo.xml`** | Служебная информация **о самой выгрузке** (версия формата дампа, перечни объектов и т.п. — зависит от версии платформы и режима выгрузки). Не путать с бизнес-логикой конфигурации; при разборе «что в конфигурации» опирайтесь в первую очередь на `Configuration.xml` и каталоги типов. |
| **Каталоги по типам** | `Catalogs/`, `Documents/`, `InformationRegisters/`, `Enums/`, `CommonModules/`, `Reports/` и другие — каждый тип объектов в своём каталоге; внутри — папки или файлы конкретных объектов. |

Имеет смысл держать в голове **двойное представление**: «логическое дерево метаданных» в платформе и **файловое дерево** на диске. Связующим звеном для **перечня типов верхнего уровня** служит блок дочерних объектов в `Configuration.xml`.

## 2. Configuration.xml: MetaDataObject и Configuration

Корневой файл обычно оформлен как **метаданные одного объекта** — конфигурации:

- Внешняя оболочка — **`MetaDataObject`** с **внутренним именем** вида `Configuration` (имя может совпадать с именем конфигурации в свойствах — ориентируйтесь на фактический XML выгрузки).
- Внутри — узлы, характерные для **корневого объекта**: прежде всего **`Properties`** и **`ChildObjects`**, плюс при необходимости другие секции по версии платформы.

Таким образом, **конфигурация целиком** в выгрузке моделируется так же, как отдельный объект метаданных, но с **особым** набором дочерних типов и свойств.

## 3. InternalInfo и ContainedObject

В выгрузках часто присутствует блок **`InternalInfo`**: служебные данные платформы (идентификаторы, версии внутренних структур, иногда перечни **`ContainedObject`** — какие вложенные сущности «содержит» данный узел с точки зрения внутреннего представления).

Для задач **обзора состава** и **редактирования прикладных объектов** обычно достаточно:

- не ломать well-formed XML и типовую вложенность;
- не удалять произвольно узлы, если не уверены в последствиях для загрузки;
- при сомнениях сверяться с **аналогом** из этой же или типовой выгрузки той же версии платформы.

Детальный смысл каждого поля `InternalInfo` — в документации к версии платформы и при необходимости в **XSD** (см. MCP в `pauk/mcp/README.md`: получение схемы для типа корня конфигурации, если инструмент это поддерживает).

## 4. Properties: ключевые поля корня

В **`Properties`** конфигурации обычно задаются параметры **целиком для всей конфигурации**. Типичные ориентиры (имена в XML могут слегка отличаться по версии — сверяйте с фактическим файлом и XSD):

| Поле (логика) | Смысл |
|-----------------|--------|
| **Name** | Внутреннее имя конфигурации. |
| **Synonym** и локализации | Представление для пользователя. |
| **Version** | Версия конфигурации (строка соглашения проекта). |
| **CompatibilityMode** | Режим совместимости **конфигурации** с версией платформы; влияет на доступные возможности языка и поведение. |
| Прочие режимы и флаги | Например параметры синхронизации, использования модальности, особенности поведения по версии — перечень зависит от платформы. |

Полный перечень и допустимые значения — **только** из схемы и документации для вашей версии; не подставляйте значения «по памяти».

## 5. ChildObjects как реестр типов

**`ChildObjects`** в `Configuration.xml` — это **реестр ссылок** на дочерние объекты метаданных **верхнего уровня**: справочники, документы, общие модули, перечисления и т.д.

- Каждая запись обычно указывает на **тип** и **имя** (или иной идентификатор в формате выгрузки) объекта, который должен иметь отражение в **соответствующем каталоге** выгрузки.
- **Добавление** нового объекта метаданных в конфигурацию в терминах файлов означает: появление узла в дереве каталогов **и** согласованная запись в `ChildObjects` (порядок записей может иметь значение для читаемости diff, но критична **полнота и согласованность**).
- **Удаление** объекта — обратная операция: убрать из `ChildObjects` и удалить каталог/файлы объекта, предварительно убедившись в отсутствии ссылок (поиск по проекту, инструменты MCP по контракту подключённого сервера).

Таким образом, **`ChildObjects` — не дубль каталогов ради каталогов**, а **канонический перечень** того, что платформа считает частью конфигурации на уровне метаданных.
