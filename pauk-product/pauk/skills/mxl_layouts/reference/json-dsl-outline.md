# JSON-DSL для черновика макета (об outline → Template.xml)

Черновой **контракт данных** для генерации XML макета **табличного документа**. Назначение — единый источник правды для скриптов или ручной доработки до финального **`Template.xml`**. Поля и вложенность могут расширяться в вашем проекте; ниже — **минимальный каркас** и правила согласованности.

## Корень документа-черновика

Логические секции верхнего уровня (имена ключей — рекомендация, не стандарт платформы):

| Ключ | Назначение |
|------|------------|
| **`columns`** | Описание колонок листа |
| **`page`** | Параметры страницы (формат, поля, ориентация) |
| **`fonts`** | Список шрифтов с атрибутами (имя, размер, жирность) |
| **`styles`** | Стили ячеек: ссылки на шрифт, границы, заливку, выравнивание |
| **`areas`** | Именованные области (`namedItem`): имя + прямоугольник в **1-based** координатах |
| **`cells`** | Сетка содержимого: координаты + тип заполнения |

## Правила координат

- **`col`**, **`row`**, ширина/высота в **`areas`** — в DSL считать **с единицы (1-based)**, как удобно разработчику для сопоставления с номерами строк в Excel и с **`ПолучитьОбласть`** по координатам в коде.
- Генератор XML переводит 1-based в формат, который требует **XSD** (если схема использует 0-based или иной базис — преобразование **только** в шаге генерации).

## `page`

- Значение по умолчанию для типовых печатных форм: формат **`A4`**, ориентация **`portrait`** или **`landscape`** — явно задавайте в DSL, чтобы генератор выставил узлы страницы в XML без догадок.

## `columns`

- Массив объектов: ширина колонки, опционально **`styleId`** — ссылка на элемент из **`styles`**.
- Число элементов = число колонок листа; порядок = слева направо.

## `fonts` и `styles`

- **`fonts`**: стабильные идентификаторы (`id`), чтобы **`styles`** ссылались на них.
- **`styles`**: каждый стиль — ссылка на шрифт, цвет текста/фона (как индекс палитры или как абстрактный токен, который генератор мапит в индексы XML — **единая** таблица соответствий в генераторе обязательна, иначе «поедут» индексы палитр).

## `areas`

- Каждая запись: **`name`** (строка для `namedItem` и **`ПолучитьОбласть`**), **`row`**, **`col`**, **`rowCount`**, **`colCount`** в 1-based системе.
- Не пересекайте семантику: одно **`name`** — одна область.

## `cells`

- Массив ячеек с полями **`row`**, **`col`** (1-based), **`styleId`**, **`fill`**.
- Поле **`fill`** — объект с полем **`type`**:

| `type` | Смысл |
|--------|--------|
| **`param`** | Имя параметра макета; в XML — узел параметра; в коде — установка через **`Параметры`** |
| **`text`** | Готовая строка в ячейке (`tl` без плейсхолдеров) |
| **`template`** | Текст с подстановками **`[Имя]`**; имена должны совпадать с параметрами |

Опционально для строк детализации — флаг или секция **`detail`**, которую генератор отражает как **`detailParameter`** в XML.

## Сборка в Template.xml

- Преобразование черновика **JSON → `Template.xml`** выполняется **вашими скриптами сборки** в репозитории **либо вручную** после прохождения черновика через проверку.
- Перед загрузкой макета в конфигурацию **проверьте XML по XSD** платформы и при доступности инструментов — через **MCP** валидации XML в среде проекта (см. **`pauk/mcp/README.md`**).
- После генерации **откройте макет** в конфигураторе и выполните тестовую печать: автоматическая генерация не заменяет визуальную проверку объединений и переносов.
