# Области, параметры и заполнение табличного документа

Материал для кода на BSL и для разметки макета. Имена методов и точные сигнатуры — в синтакс-помощнике вашей версии платформы.

## Типы областей (концепция макета)

В терминах **`ТабличныйДокумент`** и конструктора макетов области делятся по **назначению**:

| Тип | Смысл | Типичное использование |
|-----|--------|-------------------------|
| **Строки** | Область, повторяемая **по вертикали** при выводе детальных записей | Строка табличной части, строка итогов |
| **Колонки** | Повтор по **горизонтали** (редко в простых печатных формах) | Однотипные колонки по периодам и т.п. |
| **Прямоугольник** | Фиксированный фрагмент листа **без** встроенного цикла платформы по типу строк/колонок | Шапка, подвал, блок условий |
| **Рисунок** | Вставка **картинки** (логотип, штрихкод как изображение) | Оформление |

В XML макета границы областей задаются через **`namedItem`** и/или структуру строк; в коде к ним обращаются через **`ПолучитьОбласть`**.

## Имена и пересечения для `ПолучитьОбласть`

- **`ПолучитьОбласть`** возвращает фрагмент макета по **имени** именованной области или по **координатам** (в зависимости от перегрузки).
- Если в макете **несколько именованных областей**, имена должны быть **уникальны** в рамках контекста поиска (как задано платформой для данного макета).
- **Вложенность и пересечение**: область «внутри» другой может иметь **своё** имя; при выводе **сначала** получают родительский фрагмент, **потом** вставляют или заполняют дочерние. Не полагайтесь на неявное пересечение имён без проверки — лучше явно именовать шапку, строку, подвал.
- После **`Вывести`** порядок областей на странице определяется **порядком вызовов** в коде, а не только порядком строк в XML.

## Параметры по областям

- У **`ТабличныйДокумент`** (и у полученной **области**) есть коллекция **`Параметры`**: именованные значения для подстановки в ячейки типа **параметр**.
- **Устанавливайте параметры на том уровне**, который соответствует макету: часто — на всём документе до вывода, иногда — на локальной области после **`ПолучитьОбласть`**, если макет так спроектирован.
- Имена параметров в коде и в макете должны **совпадать**. Опечатка даёт **пустую ячейку** или текст без подстановки — без ошибки выполнения.
- Тип значения параметра должен быть **допустим для отображения** в ячейке (строка, число, дата и т.д. — по правилам платформы для печати).

## Шаблонные подстановки `[Имя]` в тексте

- В **текстовой** ячейке (`tl`) можно вставить плейсхолдеры вида **`[ИмяПараметра]`** (имя в квадратных скобках).
- При выводе платформа подставляет значение **одноимённого параметра**, если оно задано.
- **Правила именования**: используйте **латиницу или кириллицу** последовательно по проекту; избегайте пробелов внутри имени; не дублируйте одно имя для разных смыслов в одном макете.
- Сложные выражения в квадратных скобках **не** являются полноценной «формулой Excel» — это именно **шаблон параметров** макета 1С.

## `detailParameter` (детализация)

- **Детализация** — вывод **множества строк** по одному шаблону строки макета (область «строка детализации»).
- Ячейки в этой области помечаются как **`detailParameter`**: на каждой итерации вывода подставляются значения **текущей** детальной записи.
- Связка с кодом: цикл по данным + **`Вывести`** области детализации (или эквивалентный API печати) с заполнением **источника** детальных полей — по выбранному вами шаблону (запрос, таблица значений, объект).
- Имена полей детализации должны быть **согласованы** с тем, что вы передаёте в контекст вывода (структура, соответствие имён колонок и параметров — по соглашению проекта).

## Контрольный чеклист перед сдачей макета

- Все **`namedItem`**, вызываемые из кода, **существуют** в XML и **имеют** непустой диапазон.
- Каждый **`Параметр`** и каждый **`[Имя]`** в тексте имеет **установку** в BSL на всех ветках кода.
- Область детализации **не** содержит «забытых» **`detailParameter`** без данных в тестовом наборе.
