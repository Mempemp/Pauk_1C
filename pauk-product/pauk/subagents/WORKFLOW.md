# Цепочка субагентов

Значения для поля `route.workflow` при сборке маршрута. Порядок подключения ролей — по `pauk/routing/PROTOCOL.md` (для `trivial` цепочка выполняется в одном ответе с `route`; для `full` — начиная со следующего сообщения пользователя).

## Как оркестратор использует субагентов

- **Человек не обязан** вызывать субагента по имени: роль активируется, только если её идентификатор попал в **`route.workflow`** при шаге 1 маршрутизации (`pauk/routing/PROTOCOL.md`).
- **Навыки** для шага задаёт **`route.skills`** (тот же шаг 1, каталог **`SKILLS-CATALOG.md`**). Промпт роли в **`pauk/subagents/*.md`** читается **вместе** с телами навыков из `skills`, а не вместо них.
- При смене шага цепочки при необходимости **пересмотри** `skills` (добавь или убери `skill_id`), если контекст шага изменился; не повторяй один и тот же `skill_id` в `route.skills`.

## Идентификаторы ролей

`task-analyst` → `explore` → (при необходимости) `design` → (условно) **`metadata-manager`** → `implement` → `verify`

Промпт роли **`metadata-manager`**: файл **`metadata-manager.md`** в этом каталоге.

## Условное включение `metadata-manager`

Добавляйте **`metadata-manager`** в `route.workflow`, если выполняются **оба** условия:

1. Задача **многошаговая** по **XML-выгрузке** / **объектам метаданных** / **структуре Form.xml** и валидации (типично **`scale: full`**, либо явно согласованный с человеком полный цикл по метаданным).
2. В маршруте уже учтён навык **`metadata_manage`** (и при необходимости **`form_patterns`**) — см. сигналы в **`pauk/routing/SKILLS-CATALOG.md`**.

Если правка **точечная** (один файл, без цепочки compile → validate → fix), роль можно **не** выделять: достаточно `implement` → `verify` с теми же навыками.

## Масштаб

- **trivial:** часто `implement` → `verify`; при необходимости короткий `explore`. Роль **`metadata-manager`** обычно **не** включают.
- **full:** с `task-analyst` в начале; `design` — при крупном проектировании реализации (условия принятия blueprint — `pauk/policy/A-INVARIANTS.md`). При тяжёлой работе по выгрузке — **`metadata-manager`** после `design` или вместо части шагов `implement`, если основной объём — XML, а не BSL.

## Предопределённые сценарии (пресеты)

**Зачем:** снять неоднозначность **большой** задачи и задать **базовый** порядок ролей и handoff. Готовый файл сценария в репозитории: **`.cursor/commands/flow-full-pipeline.md`** (там, где среда поддерживает вызов из этой папки — через палитру; иначе открывать вручную). **Обычный** режим отдельной **команды** не требует. Пока пресетов мало; новые сценарии — **строкой в таблицу** ниже + при необходимости ещё один `flow-*.md` в `.cursor/commands/`, без плоджения `*.md` в `pauk/subagents/`, пока не нужна **новая** роль.

| id пресета | Команда Cursor | `scale` | Суть |
|------------|----------------|---------|------|
| `default` | *(нет — обычный ввод)* | `trivial` по умолчанию, `full` по явным сигналам из `PROTOCOL.md` | Как **§ Идентификаторы ролей** выше, без жёсткой фиксации handoff-файла. |
| `trivial-progressive` | *нет (режим по умолчанию)* | `trivial` (если явный сигнал объёма у пользователя не сильнее, чем в `PROTOCOL`) | Одно содержательное сообщение, прогрессивный подбор `route.skills`, цепочка обычно `implement` → `verify`, при необходимости короткий `explore`. Опциональные **гейты** по сигналам: `code-review` после крупных правок, `code-refactor` при заявленной зачистке без смены поведения, `error-fix` при фокусе на ошибке — см. **§ Дополнительные роли** и `SKILLS-CATALOG.md`. **Артефакта** маршрутизатора не требуется, если другое не согласовано. |
| `full-pipeline` | `flow-full-pipeline` (`.cursor/commands/`) | `full` | **Объёмный** контур: **изоляция** по фазам через артефакт; см. **Базовая цепочка** ниже. |

### `full-pipeline` — базовая цепочка (значения для `route.workflow` по умолчанию)

Ниже — **ориентир** для `ARTIFACT-ROUTER.md` и `task-analyst` (разрешён пропуск шагов с явной фиксацией в шаблоне, см. `A-INVARIANTS` про blueprint и scope):

1. **`task-analyst`** — выход: заполненный **`pauk/routing/ARTIFACT-ROUTER.md`**, в т.ч. `route.skills` и `route.workflow` на весь план, вопросы в §6 шаблона. Один **ограниченный** фокус, без реализации кода.
2. **`explore`** — разведка, список затронутого, гипотезы; без лишнего handoff, если `ARTIFACT-ROUTER` достаточен; иначе продолжение в том же духе шаблона.
3. **`solution-architect`** — по сигналу крупного изменения границ/контракта; можно **слить** с `design` в одной фазе, если handoff в артефакте это отражает.
4. **`design`** — крупноблочный blueprint, если по политике требуется; принятие человеком — `A-INVARIANTS.md`.
5. **Условно `metadata-manager`** — по **§ Условное включение** выше.
6. **`implement`** — правки по плану.
7. **`code-review`** — **отдельный** смысловой заход (свежий угол ревью); `route.skills` минимум `coding-standards` + сигналы. Имеет смысл **после** существенных диффов, не обязан буквально в новом «окне» Cursor — но с **сужением** передаваемого контекста к сводке + диффу.
8. **`verify`** — чеклисты, линтер, согласованность.
9. **`task-analyst`** (повторно) — **режим приёмки:** сверка сделанного с §2–4 **исходного** `ARTIFACT-ROUTER` и `A-INVARIANTS`, без дублирения полноценного `implement`; новая роль в `pauk/subagents/*.md` **не** требуется, контракт — в handoff-таблице плана (если в проекте приняли отдельное имя, зафиксируйте в `route.workflow` как `task-analyst` с пометой в чате «приёмка»).

**Масштабирование:** отдельные **вариации** `full-pipeline` (например, без `solution-architect` или с ранним `arch-review`) оформляйте **копированием** строки в таблицу пресетов и, при необходимости, новой `flow-*.md` командой, чтобы не раздувать один файл команды.

## Дополнительные роли (BL-012, опционально в `route.workflow`)

Идентификатор в `workflow` | Файл промпта | Когда добавлять
---------------------------|--------------|-----------------
`solution-architect` | `solution-architect.md` | Крупная доработка: нужен явный архитектурный контракт до/параллельно `design`
`arch-review` | `arch-review.md` | Крупное изменение: отдельный архитектурный гейт
`code-review` | `code-review.md` | После существенных правок BSL — отдельный проход ревью
`code-refactor` | `code-refactor.md` | Запланированная зачистка техдолга без смены поведения
`error-fix` | `error-fix.md` | Узкий цикл по ошибкам компиляции/рантайма
`external-docs` | `external-docs.md` | Внешняя документация (гайды, codemap, API) — при согласованной структуре каталогов в проекте
`deploy-test` | `deploy-test.md` | Деплой в тестовую ИБ и смоук в веб-клиенте — **только** по регламенту и путям из репозитория проекта
`performance-opt` | `performance-opt.md` | Производительность и запросы — после явных симптомов или как отдельный гейт

### Ожидаемые `route.skills` вместе с ролями

На шаге 1 согласуйте **`route.skills`** с выбранной ролью (каталог — **`pauk/routing/SKILLS-CATALOG.md`**, сигналы — тот же файл):

| Роль | Как правило в `route.skills` |
|------|------------------------------|
| `code-refactor` | Как минимум **`coding-standards`**. При **существенной** перестройке кода и/или метаданных без смены поведения — также **`refactor_workflow`** (методология top-down / bottom-up и порядок инструментов в **`pauk/mcp/README.md`**). Остальные id — по сигналам (например **`query-patterns`**, **`form_patterns`**, **`metadata_manage`**, **`error_handling`**). |
| `solution-architect`, `arch-review` | База по задаче из каталога. Если в фокусе **обмен с внешней системой**, **HTTP/REST**, прототип контура **вне клиента 1С** — добавьте **`integration_patterns`**; при **внешних вызовах с записью и сбоями** — обычно и **`error_handling`**. |
| `code-review` | Как минимум **`coding-standards`**. Если ревью **после** крупного рефакторинга — убедитесь, что в маршруте был или остаётся релевантный набор (часто **`refactor_workflow`** + узкие по сигналам). |

Для каждой роли на шаге 1 задайте **`route.skills`** (см. **`pauk/routing/SKILLS-CATALOG.md`**). Сводное сопоставление исходных агентов Comol с этими ролями ведётся **в репозитории фабрики** (задача BL-012); в прикладной копии пакета достаточно этой таблицы и промптов в `pauk/subagents/`.

Промпты ролей — файлы в этом каталоге. Базовая цепочка без тяжёлых метаданных часто остаётся на внешних промптах `task-analyst` / `explore` / `design` / `implement` / `verify`; перечисленные выше файлы — расширения под маршрутизацию.
