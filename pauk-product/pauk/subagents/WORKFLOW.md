# Цепочка субагентов

Значения для поля `route.workflow` при сборке маршрута. Порядок подключения ролей — по **`pauk/routing/PROTOCOL.md`**. **Режим сообщений** (всё в одном ответе vs короткий первый ответ, затем со следующего сообщения пользователя) задаётся **пресетом** — таблица ниже; без команды из **`.cursor/commands/`** — одно содержательное сообщение.

## Обычный ввод (без команды из `.cursor/commands/`)

Без команды из **`.cursor/commands/`** основная нагрузка на **`route.skills`** и **`SKILL.md`**. **Не** расширять **`route.workflow`** ролями из § **«Дополнительные роли»**, если в тексте задачи или у человека нет на то явного основания. Промпты **`pauk/subagents/*.md`** **не** просматривать «на всякий случай» — см. **`PROTOCOL.md`**, § «Субагенты и навыки».

## Как оркестратор использует субагентов

- **Человек не обязан** называть субагента по имени: идентификатор роли попадает в **`route.workflow`** на шаге 1 по **`PROTOCOL.md`** / пресету **или** человек **явно** просит режим, соответствующий файлу в **`pauk/subagents/`**.
- **Навыки** для шага задаёт **`route.skills`** (тот же шаг 1, каталог **`SKILLS-CATALOG.md`**). Файл **`pauk/subagents/<роль>.md`** читать **только если** он **существует** и роль — **текущий** шаг в **`route.workflow`** (или сработала явная просьба человека); чтение **после** тел **`SKILL.md`** из **`route.skills`**, а не вместо них.
- При смене шага цепочки при необходимости **пересмотри** `skills` (добавь или убери `skill_id`), если контекст шага изменился; не повторяй один и тот же `skill_id` в `route.skills`.

## Идентификаторы ролей

`task-analyst` → `explore` → (при необходимости) `design` → (условно) **`metadata-manager`** → `implement` → `verify`

Промпт роли **`metadata-manager`**: файл **`metadata-manager.md`** в этом каталоге.

## Условное включение `metadata-manager`

Добавляйте **`metadata-manager`** в `route.workflow`, если выполняются **оба** условия:

1. Задача **многошаговая** по **XML-выгрузке** / **объектам метаданных** / **структуре Form.xml** и валидации в пресетах **`full-pipeline`** / **`technical-design`**, либо при **явно** согласованном с человеком полном цикле по метаданным в обычном вводе.
2. В маршруте уже учтён навык **`metadata_manage`** (и при необходимости **`form_patterns`**) — см. сигналы в **`pauk/routing/SKILLS-CATALOG.md`**.

Если правка **точечная** (один файл, без цепочки compile → validate → fix), роль можно **не** выделять: достаточно `implement` → `verify` с теми же навыками.

## Предопределённые сценарии (пресеты)

**Зачем:** задать **базовый** порядок ролей и handoff для объёмного контура. Готовые файлы: **`.cursor/commands/flow-full-pipeline.md`**, **`.cursor/commands/flow-technical-design.md`** (палитра команд или открыть вручную). Обычный ввод отдельной команды **не** требует. Новые сценарии — строка в таблицу + при необходимости **`flow-*.md`**; новые роли — **`pauk/subagents/<роль>.md`**, если нужен новый идентификатор в **`route.workflow`**.

Объёмные пресеты (**`full-pipeline`**, **`technical-design`**): **`route.skills`** на шаге 1 подбираются как при обычном вводе; отличаются **режим первого ответа** (колонка ниже) и полная цепочка **`route.workflow`**.

| id пресета | Команда Cursor | Режим первого ответа | Суть |
|------------|----------------|----------------------|------|
| `default` | *(нет — обычный ввод)* | Одно сообщение с работой по задаче | **`route.skills`** по **`SKILLS-CATALOG.md`**, короткий **`route.workflow`** (часто `implement` → `verify`, при необходимости `explore`); как **§ Идентификаторы ролей** выше. Промпты **`pauk/subagents/*.md`** и **`ARTIFACT-ROUTER`** — по явной просьбе человека или если текущий шаг **`route.workflow`** имеет файл роли. Опциональные роли — **§ Дополнительные роли** и каталог. |
| `full-pipeline` | `flow-full-pipeline` | Короткий первый ответ **без** тел **`SKILL.md`**; со **следующего** сообщения пользователя — пошагово по **`route.workflow`**, тела навыков по мере шагов | Объёмный контур и handoff по фазам — § **Базовая цепочка** ниже. |
| `technical-design` | `flow-technical-design` | Как у **`full-pipeline`** | ТЗ в **`Technical-Design/`** — § **`technical-design`** ниже. |

### `full-pipeline` — базовая цепочка (значения для `route.workflow` по умолчанию)

Ниже — **ориентир** для `ARTIFACT-ROUTER.md` и `task-analyst` (разрешён пропуск шагов с явной фиксацией в шаблоне, см. `A-INVARIANTS` про blueprint и scope):

1. **`task-analyst`** — выход: заполненный **`pauk/routing/ARTIFACT-ROUTER.md`**, в т.ч. `route.skills` и `route.workflow` на весь план, вопросы в §6 шаблона. Один **ограниченный** фокус, без реализации кода; промпт роли — **`task-analyst.md`**.
2. **`explore`** — разведка, список затронутого, гипотезы; без лишнего handoff, если `ARTIFACT-ROUTER` достаточен; иначе продолжение в том же духе шаблона.
3. **`solution-architect`** — по сигналу крупного изменения границ/контракта; можно **слить** с `design` в одной фазе, если handoff в артефакте это отражает.
4. **`design`** — крупноблочный blueprint, если по политике требуется; принятие человеком — **`A-INVARIANTS.md`**; промпт роли — **`design.md`**.
5. **Условно `metadata-manager`** — по **§ Условное включение** выше.
6. **`implement`** — правки по плану.
7. **`code-review`** — **отдельный** смысловой заход (свежий угол ревью); `route.skills` минимум `coding-standards` + сигналы. Имеет смысл **после** существенных диффов, не обязан буквально в новом «окне» Cursor — но с **сужением** передаваемого контекста к сводке + диффу.
8. **`verify`** — чеклисты, линтер, согласованность.
9. **`task-analyst`** (повторно) — **режим приёмки:** сверка сделанного с §2–4 **исходного** `ARTIFACT-ROUTER` и `A-INVARIANTS`, без дублирения полноценного `implement`; тот же идентификатор и промпт **`task-analyst.md`**, смысл шага — «приёмка» (пометка в чате при необходимости); отдельный **новый** файл роли не нужен.

**Масштабирование:** отдельные **вариации** `full-pipeline` (например, без `solution-architect` или с ранним `arch-review`) оформляйте **копированием** строки в таблицу пресетов и, при необходимости, новой `flow-*.md` командой, чтобы не раздувать один файл команды.

### `technical-design` — ТЗ и нарезка на задачи (без реализации)

Цель: по описанию человека получить **зафиксированное ТЗ** и **список задач** в виде файлов в **`Technical-Design/`**; цепочка **заканчивается** на материализации артефактов (дальше — на усмотрение человека).

**Ориентир для `route.workflow`** (два захода **`task-analyst`** с разным смыслом — помечайте в чате «аналитика» / «фиксация ТЗ» при необходимости):

1. **`task-analyst`** — вопросы, **corner cases**, крупная логика; выход: согласованный с человеком **`ARTIFACT-ROUTER.md`** (или его разделы 2–6). **Стоп** до явного подтверждения логики.
2. **`solution-architect`** — крупноблочная архитектура; **стоп** до явного согласования.
3. **`design`** — **опционально:** спросить человека, нужен ли blueprint; при **да** — согласовать blueprint (**`design.md`**); при **нет** — зафиксировать пропуск.
4. **`task-analyst`** (повторно) — итог, нарезка на задачи, создание **`Technical-Design/`** по **`pauk/routing/TECHNICAL-DESIGN-SCAFFOLD.md`** (`README.md` эпика + подпапки **`Txx-slug/`** с **`task.md`** и **`solution-design.md`**).

Подробный порядок и ссылки — **`.cursor/commands/flow-technical-design.md`**.

| Идентификатор в `workflow` | Файл промпта | Когда добавлять |
|----------------------------|--------------|-----------------|
| `solution-architect` | `solution-architect.md` | Крупная доработка: нужен явный архитектурный контракт до/параллельно `design` |
| `arch-review` | `arch-review.md` | Крупное изменение: отдельный архитектурный гейт |
| `code-review` | `code-review.md` | После существенных правок BSL — отдельный проход ревью |
| `code-refactor` | `code-refactor.md` | Запланированная зачистка техдолга без смены поведения |
| `error-fix` | `error-fix.md` | Узкий цикл по ошибкам компиляции/рантайма |

### Ожидаемые `route.skills` вместе с ролями

На шаге 1 согласуйте **`route.skills`** с выбранной ролью (каталог — **`pauk/routing/SKILLS-CATALOG.md`**, сигналы — тот же файл):

| Роль | Как правило в `route.skills` |
|------|------------------------------|
| `code-refactor` | Как минимум **`coding-standards`**. При **существенной** перестройке кода и/или метаданных без смены поведения — также **`refactor_workflow`** (методология top-down / bottom-up и порядок инструментов в **`pauk/mcp/README.md`**). Остальные id — по сигналам (например **`query-patterns`**, **`form_patterns`**, **`metadata_manage`**, **`error_handling`**). |
| `solution-architect`, `arch-review` | База по задаче из каталога. Если в фокусе **обмен с внешней системой**, **HTTP/REST**, прототип контура **вне клиента 1С** — добавьте **`integration_patterns`**; при **внешних вызовах с записью и сбоями** — обычно и **`error_handling`**. |
| `code-review` | Как минимум **`coding-standards`**. Если ревью **после** крупного рефакторинга — убедитесь, что в маршруте был или остаётся релевантный набор (часто **`refactor_workflow`** + узкие по сигналам). |
| `task-analyst` (пресет **`technical-design`**, первый заход) | **`coding-standards`** и узкие навыки по сигналам из **`SKILLS-CATALOG.md`** (часто **`integration_patterns`**, **`form_patterns`**, **`metadata_manage`** — по контексту эпика). |
| `design` (после **`solution-architect`**) | База из каталога по сигналам; при крупном контракте с внешними системами — **`integration_patterns`**, **`error_handling`**. |

Для каждой роли на шаге 1 задайте **`route.skills`** (см. **`pauk/routing/SKILLS-CATALOG.md`**). Для работы в прикладном репозитории достаточно таблиц выше и промптов в **`pauk/subagents/`**.

Промпты с отдельным файлом: § **«Дополнительные роли»** и таблица в **`README.md`** этого каталога, плюс **`task-analyst.md`** и **`design.md`** (базовые идентификаторы с промптом в пакете). Для **`explore`**, **`implement`**, **`verify`** одноимённых **`.md`** **нет** — семантика в **`WORKFLOW.md`** и **`PROTOCOL.md`**. Для **`task-analyst`** и **`design`** читайте соответствующие **`.md`** вместе с **`SKILL.md`** при активном шаге в **`route.workflow`**. Для **`task-analyst`** дополнительно **`pauk/routing/ARTIFACT-ROUTER.md`**; для финала **`technical-design`** — **`pauk/routing/TECHNICAL-DESIGN-SCAFFOLD.md`**.
