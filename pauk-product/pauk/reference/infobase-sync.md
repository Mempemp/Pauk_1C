# Выгрузка и загрузка конфигурации (ИБ ↔ каталог проекта)

Операционный цикл **вне** тела навыков: пути, версия платформы и учётные данные задаёт **репозиторий проекта** (скрипты, `infobasesettings.md`, CI), не оснастка Pauk. Не подставляйте пути и имена с «чужой» машины.

**Связь с оснасткой:** правка XML в каталоге — [metadata_manage/SKILL.md](../skills/metadata_manage/SKILL.md), [form_patterns/SKILL.md](../skills/form_patterns/SKILL.md). Поиск и проверка кода/метаданных — [mcp/README.md](../mcp/README.md). Смоук и роль в маршруте — [subagents/deploy-test.md](../subagents/deploy-test.md), [subagents/WORKFLOW.md](../subagents/WORKFLOW.md).

## 1. Локальные договорённости

- Файл **`infobasesettings.md`** (или аналог в вашем репо), если команда его использует: строка подключения к **тестовой** ИБ (`/F` для файловой, `/S` для серверной), при необходимости URL публикации веб-клиента для проверок, путь к логу обновления. **Секреты** — только в хранилище/файлах проекта, не в чате.
- Путь к **`1cv8.exe`**: фактическая версия платформы в вашей среде.
- **Корень выгрузки** в репозитории: тот каталог, который передаёте в `-LoadConfigFromFiles` / `-DumpConfigToFiles` (часто корень репо или `src/`, как принято у команды).

## 2. Загрузка каталога в информационную базу

Типовой порядок после правок в файлах выгрузки:

1. **Загрузка конфигурации из файлов** — `DESIGNER` / `IBConnection` / `LoadConfigFromFiles` с корнем выгрузки, лог в файл.
2. Пауза **несколько секунд**, чтение лога на ошибки.
3. **Обновление структуры БД** — `UpdateDBCfg` с параметрами по регламенту команды (в т.ч. `-Dynamic+`, при необходимости `-SessionTerminate force`).

Синтаксис **PowerShell** (кавычки, `;` вместо `&&`, `Invoke-WebRequest` и т.д.) — навык [../skills/powershell_windows/SKILL.md](../skills/powershell_windows/SKILL.md), не BSL.

Шаблон PowerShell (подставьте свои значения):

```powershell
& '<PATH_1CV8>\1cv8.exe' DESIGNER <IB_MODE> <IB_STRING> <DisableStartupMessages> /LoadConfigFromFiles <PROJECT_ROOT> /Out <LOG_PATH>
# … пауза, проверка <LOG_PATH> …
& '<PATH_1CV8>\1cv8.exe' DESIGNER <IB_MODE> <IB_STRING> <DisableStartupMessages> /UpdateDBCfg -Dynamic+ -SessionTerminate force /Out <LOG_PATH>
```

- `<IB_MODE>`: для файловой базы вместе с `IB_STRING` оформляйте по документации платформы, например `/F 'C:\...'`; для серверной — `/S` и имя.
- Смена пользователя и пароля для **Designer** — по политике проекта (`/N`, `/P` и т.д. — только из ваших шаблонов, не копируйте чужие учётки).

## 3. Выгрузка выбранных объектов в каталог (репозиторий)

- Подготовьте список объектов **`repoobjects.txt`** (формат по [ITS / документации](https://its.1c.ru) для вашей версии) — **полные** имена, без лишнего вложенного каталога выгрузки, если команда договаривается **строго** в «текущий» корень.
- **DumpConfigToFiles** с флагом `-listFile` на этот список; при доработке через расширение — уточните **`-Extension <Имя>`** в регламенте. Лог — в согласованный файл.
- **Имена инструментов MCP** для обхода метаданных и поиска объектов не перечисляйте «с памяти» — [mcp/README.md](../mcp/README.md); приоритет: специализированные серверы, затем чтение файлов.

## 4. Проверка в веб-клиенте (смоук)

URL и сценарий (какой документ/форма) — из **задачи** и **файлов проекта** (`infobasesettings` или вики). Семантически согласуется с субагентом **`deploy-test`**: пути и команды **не** выдумывайте, если в репозитории нет регламента.

## 5. Команды Cursor (опционально)

В поставке могут лежать тонкие файлы **`.cursor/commands/*.md`**, ведущие в этот документ — **ускорители** для человека, не замена **`PROTOCOL.md`**.
