## Шаг 1 — маршрут (`route`) для исполнителя

Перед выбором навыков использовать **только** таблицу `skill_id` из **`SKILLS-CATALOG.md`** в этой папке; итоговый набор **`skills`** — только перечисленные там идентификаторы, без выдуманных имён.

### Сигналы: тип модуля и контекст задачи

Перед фиксацией **`route.skills`** оцени, **какие модули и файлы** затрагивает задача: явные пути в сообщении, вложения, `@`-ссылки, имена файлов в цитатах, при наличии — открытые вкладки IDE (если платформа передала их в контекст). Подсказки «какой узкий навык почти всегда уместен» — в **`SKILLS-CATALOG.md`**, раздел **«Сигналы → навыки»** (новых `skill_id` раздел **не** вводит). Если сигналов нет — достаточно **`coding-standards`** и смысла задачи.

Исполнитель **обязан** внутренне зафиксировать маршрут в структуре **`route`**, **как если** он записан в YAML — даже если в ответ пользователю YAML **не** попадает.

```yaml
route:
  workflow: []
  skills: []
  blockers: []
```

После фиксации **`route.skills`** внутри **`route`:** по **`MCP-CATALOG.md`** в этой папке оценить **«Сигналы → mcp_id»**. Для каждого подошедшего **`mcp_id`**: если в **схеме MCP сессии** нет инструментов этого сервера — не читать **`pauk/mcp/route-*.md`** для этого id; если **есть** — прочитать **`pauk/mcp/route-<mcp_id>.md`** и следовать. Иначе шаг для MCP пропустить.

### Субагенты и навыки

**Источники полей:** **`route.workflow`** и **`route.skills`** согласуются с этим файлом, **`pauk/subagents/WORKFLOW.md`** и, при сценарии из команды, файлом **`.cursor/commands/*.md`**. Отдельного **`skill_id`** «маршрутизация» в **`SKILLS-CATALOG.md`** нет; тематические знания — только **`pauk/skills/<skill_id>/SKILL.md`**.

**Режим сообщений** (одно содержательное сообщение целиком vs короткий первый ответ, затем со следующего сообщения пользователя) для объёмных пресетов описан **только** в **`WORKFLOW.md`** (таблица пресетов, § **`full-pipeline`**, **`technical-design`**).

**Проактивный обход `pauk/subagents/*.md` запрещён.** Роли подключает оркестратор (исполнитель по этому файлу и **`.cursor/rules/pauk-entry.mdc`**): фиксация **`route.workflow`** по **`WORKFLOW.md`** и сигналам задачи **или** **явная** просьба человека сослаться на конкретный файл в **`pauk/subagents/`**.

**Обычный ввод** (нет сценария из `.cursor/commands/`): опора на **`route.skills`** и соответствующие **`SKILL.md`**. Короткий **`route.workflow`** (часто `implement` → `verify`, иногда `explore`) — **метки этапов** по **`WORKFLOW.md`**, не повод искать несуществующие файлы промптов. Роли из § **«Дополнительные роли»** в **`WORKFLOW.md`** в **`route.workflow` не добавлять** без явного смысла в задаче или явной просьбы человека.

**Когда читать файл роли `pauk/subagents/<роль>.md`:** только если человек **явно** попросил эту роль **или** идентификатор — **текущий** шаг **`route.workflow`** и для него **существует** **`pauk/subagents/<роль>.md`**. Порядок: **сначала** все **`pauk/skills/<skill_id>/SKILL.md`** из **`route.skills`**, затем файл роли (роль **поверх** уже согласованных навыков).

**Идентификаторы и вспомогательные шаблоны:**

- **`task-analyst`**, **`design`** — в пакете **`task-analyst.md`**, **`design.md`**; при активном шаге читать **после** **`SKILL.md`** из **`route.skills`**. Для **`task-analyst`** в объёмных пресетах дополнительно **`pauk/routing/ARTIFACT-ROUTER.md`**; для финала пресета **`technical-design`** — **`pauk/routing/TECHNICAL-DESIGN-SCAFFOLD.md`**.
- **`explore`**, **`implement`**, **`verify`** — **отдельных** одноимённых **`.md`** в поставке **нет**; семантика шага — в **`WORKFLOW.md`** и общие правила здесь.

**Условное `metadata-manager`:** критерии в **`WORKFLOW.md`**, § «Условное включение».

### Видимость для пользователя

**По умолчанию** человеку **не** показывать блок YAML `route`: служебная разметка для исполнителя, не ответ по задаче 1С.

Допустимо **одно короткое** нейтральное предложение (без обязательного перечисления `skill_id`) или **сразу** полезный текст по задаче.

**Вывести YAML `route` в чат** — если пользователь **явно** попросил (маршрут, route, отладка оснастки, «покажи skills») или после **словесной** просьбы поправить навыки/цепочку.

### Исполнение и последующие шаги

**Обычный ввод** (нет активного пресета из команды): в **том же** сообщении прочитать каждый **`SKILL.md`** по `skills`, дальше работать по **`route.workflow`**; ограничения из **`pauk/mcp/route-*.md`**, прочитанных на шаге 1, соблюдать в ответе. Без проактивного чтения **`pauk/subagents/*.md`** (см. § выше). Полный формальный артефакт **`task-analyst`** не обязателен; **нельзя** закончить без содержательного ответа по задаче.

**Пресет из команды** (файл **`.cursor/commands/*.md`**): первый ответ, тела навыков и пошаговое исполнение — **строго** по **`WORKFLOW.md`** для выбранного пресета (здесь не дублируется).

**После короткого первого ответа** (пресеты с поэтапным стартом, см. **`WORKFLOW.md`**): отдельное подтверждение набора навыков человеком **не** требуется — **`route.skills`** уже задан на шаге 1. Продолжение — **следующее сообщение** пользователя (напр. «продолжай», уточнение задачи, правка границ scope); человек **может** словами попросить изменить навыки или цепочку — скорректировать внутренний **`route`**. Два шага разделяют лёгкое планирование и тяжёлый аналитический слой.

**После продолжения диалога:** при **обычном вводе** работа по задаче уже в первом содержательном сообщении — дальше обычный диалог, **`pauk/policy/A-INVARIANTS.md`**. При **пресете с поэтапным стартом**: после следующего сообщения пользователя при необходимости снова **`SKILLS-CATALOG.md`** и **`MCP-CATALOG.md`** (с шагом MCP по этому файлу и **`pauk/mcp/route-*.md`** при наличии серверов), затем **пошагово** по **`route.workflow`** — для шага с **`pauk/subagents/<роль>.md`** порядок чтения как в § «Когда читать файл роли»; для **`explore`**, **`implement`**, **`verify`** (нет файла промпта) — только **`WORKFLOW.md`**; для **`task-analyst`**, **`design`** — их **`.md`** в **`pauk/subagents/`** плюс **`WORKFLOW.md`**. Политика — **`pauk/policy/A-INVARIANTS.md`**. Шаблоны **`ARTIFACT-ROUTER.md`** и **`TECHNICAL-DESIGN-SCAFFOLD.md`** — по условиям пресета в **`WORKFLOW.md`** (команды **`flow-full-pipeline`**, **`flow-technical-design`**).

**Навыки из `route.skills`:** для каждого `skill_id` читать **`pauk/skills/<skill_id>/SKILL.md`** (см. «Где лежит тело навыка» в **`SKILLS-CATALOG.md`**); прочие файлы того же навыка — только по ссылкам из **`SKILL.md`**, без угадывания путей.
