# Точка входа и шаг 1 маршрутизации

## Вход задачи


| Поле     | Обязательность |
| -------- | ---------------- |
| Задача   | да               |
| Не делаем | нет             |
| Среда (без секретов) | желательно |
| Явный сигнал **крупной** задачи | нет; см. ниже |

### Масштаб без вопроса пользователю

**По умолчанию** задача считается **локальной** → в `route` ставится `scale: trivial`. **Не задавать** отдельный вопрос «мелкая или нетривиально», если человек сам не просит помочь оценить объём.

**`scale: full`** — только если пользователь **явно** обозначил крупный объём или полный цикл работы. Достаточные примеры формулировок: объёмная / большая задача, нетривиально, полный цикл, с нуля, нужен анализ/дизайн/blueprint на всю фичу, затрагиваем много модулей / интеграция / миграция данных и т.п. Отсутствие таких сигналов → **`trivial`**, даже если задача на самом деле окажется сложной (уточнения — в ходе работы).

Явные слова «мелкая правка» / «мелко» → как правило `scale: trivial`; «нетривиально» в сообщении пользователя → как правило `scale: full`.

## Шаг 1 — маршрут (`route`) для исполнителя

Перед выбором навыков использовать **только** таблицу `skill_id` из `SKILLS-CATALOG.md` в этой папке (при необходимости прочитать файл); итоговый набор `skills` — только перечисленные там идентификаторы, без выдуманных имён.

### Сигналы: тип модуля и контекст задачи

Перед фиксацией `route.skills` оцени, **какие модули и файлы** затрагивает задача: явные пути в сообщении, вложения, `@`-ссылки на файлы, имена файлов в цитатах, при наличии — открытые в IDE вкладки (если платформа передала их в контекст). Подсказки «какой узкий навык почти всегда уместен» — в **`SKILLS-CATALOG.md`**, раздел **«Сигналы → навыки»**. Раздел **не** добавляет новых идентификаторов; он снижает риск пропустить **form_patterns**, **query-patterns** и др., когда контекст однозначно форменный или запросный. Если сигналов нет — достаточно **`coding-standards`** и смысла задачи.

Исполнитель **обязан** явно для себя зафиксировать маршрут в структуре **`route`** (поля ниже), **как если** он записан в YAML — даже если в ответ пользователю YAML **не** попадает.

Структура (внутренний контракт):

```yaml
route:
  scale: trivial | full
  workflow: []
  skills: []
  blockers: []
```

### Субагенты и навыки: кто включает роли

**Классификация объёма** (`scale`), поля **`route.workflow`** и **`route.skills`** задаются **этим файлом** совместно с **`pauk/subagents/WORKFLOW.md`** и, при явном сценарии, пресетом из **`.cursor/commands/*.md`**. Отдельного **`skill_id`** «маршрутизация» в **`SKILLS-CATALOG.md`** нет: тематические знания — только в **`pauk/skills/<skill_id>/SKILL.md`**.

Роли с промптами **`pauk/subagents/<роль>.md`** **не предназначены** для самовольного **обхода каталога** в поиске «что ещё подключить». Подключает их **оркестратор** (исполнитель по **`PROTOCOL.md`** и **`.cursor/rules/pauk-entry.mdc`**): фиксация **`route.workflow`** по **`WORKFLOW.md`** и сигналам задачи **или** **явная просьба человека** сослаться на конкретную роль из **`pauk/subagents/`** (как на отдельный режим работы).

#### Обычная задача (`scale: trivial`) без явного сценария из `.cursor/commands/`

Опора на **`route.skills`** и соответствующие **`SKILL.md`**. Короткая цепочка **`route.workflow`** (часто `implement` → `verify`, иногда с `explore`) — **метки этапов** по **`WORKFLOW.md`**, а не повод искать несуществующие файлы промптов. Роли из § **«Дополнительные роли»** в **`WORKFLOW.md`** в **`route.workflow` не добавлять** без явного смысла в задаче или явной просьбы человека. Файлы **`pauk/subagents/<роль>.md`** **не** читать проактивно: только если человек **явно** попросил эту роль **или** текущий шаг в **`route.workflow`** — идентификатор, для которого **есть** **`pauk/subagents/<роль>.md`**; тогда **сначала** все **`pauk/skills/<skill_id>/SKILL.md`** из **`route.skills`**, затем этот **`.md`**.

#### Идентификаторы ролей и файлы промптов

- **`task-analyst`**, **`design`** — в пакете есть **`pauk/subagents/task-analyst.md`** и **`pauk/subagents/design.md`**; при активном шаге в **`route.workflow`** читать их **после** **`SKILL.md`** из **`route.skills`**. Для **`task-analyst`** дополнительно шаблон **`pauk/routing/ARTIFACT-ROUTER.md`**; для финала пресета **`technical-design`** — **`pauk/routing/TECHNICAL-DESIGN-SCAFFOLD.md`**.
- **`explore`**, **`implement`**, **`verify`** — в **`WORKFLOW.md`** и здесь; **отдельных** одноимённых **`.md`** в поставке **нет**, поведение шага — по **`WORKFLOW.md`**.

#### Роль с файлом промпта в пакете

Если для **текущего** шага **`route.workflow`** существует **`pauk/subagents/<роль>.md`**: **сначала** прочитать все **`pauk/skills/<skill_id>/SKILL.md`** из **`route.skills`**, затем этот файл — роль идёт **поверх** уже согласованных навыков.

Соответствие: `scale: trivial` → короткий **`route.workflow`** из **`WORKFLOW.md`** (см. ограничения выше); `scale: full` → цепочка оттуда же. Условное **`metadata-manager`** — в том же **`WORKFLOW.md`**.

### Видимость для пользователя

**По умолчанию** человеку **не** показывать блок YAML `route`: это служебная разметка для дисциплины исполнителя, а не содержательный ответ по задаче 1С.

Допустимо **одно короткое** нейтральное предложение человеческим языком (без обязательного перечисления `skill_id`), например: «Опираюсь на типовые правила по формам и сразу разберу вопрос.» Или **вообще без** вступления — сразу полезный текст по задаче.

**Вывести YAML `route` в чат** — если пользователь **явно** попросил (маршрут, route, отладка оснастки, «покажи skills») или после **словесной** просьбы поправить навыки/цепочку — по желанию одним блоком для ясности.

### Поведение после фиксации маршрута

**`scale: trivial`:** в **том же** сообщении выполнить **шаг 2** — не ждать следующего сообщения пользователя. Прочитать **`SKILL.md`** по каждому id из **`skills`**, дальше работать по задаче и **`route.workflow`** (метки этапов; см. § «Субагенты и навыки» про отсутствие проактивного чтения **`pauk/subagents/*.md`**). Полный формальный артефакт **`task-analyst`** не обязателен; главное — не обрывать ответ на «пустом» маршруте без ответа по задаче.

**`scale: full`:** в этом сообщении для человека — **короткий** осмысленный текст (1–2 предложения): что будет дальше и что написать в ответ („продолжай“ или уточнение). **Без** развёрнутой реализации и без подключения тел навыков в этом же сообщении. YAML `route` в чат — только по правилам видимости выше.

### Переход к шагу 2 при `scale: full`

**Не требуется** отдельное подтверждение или выбор навыков человеком: набор `route.skills` уже задан на шаге 1.

«Продолжение» — **следующее сообщение пользователя** после ответа с короткой подсказкой (например «продолжай», уточнение по задаче, правка границ scope). Человек **может** попросить изменить навыки / цепочку / масштаб словами — скорректировать внутренний `route` и работать по обновлённому маршруту. Два шага для **full** нужны, чтобы не смешивать в одном сообщении лёгкое планирование и тяжёлый аналитический слой.

## Шаг 2 и далее

**Если шаг 2 уже выполнен в том же сообщении** (`scale: trivial`) — дальше обычный диалог по задаче, политика `pauk/policy/A-INVARIANTS.md`.

**Если был `scale: full`:** после следующего сообщения пользователя — при необходимости снова открыть **`SKILLS-CATALOG.md`**, затем **пошагово** идти по **`route.workflow`**: для шага с существующим **`pauk/subagents/<роль>.md`** — порядок чтения как в § «Субагенты и навыки»; для **`explore`**, **`implement`**, **`verify`** (файла промпта нет) — только **`WORKFLOW.md`**; для **`task-analyst`** и **`design`** — соответствующие **`.md`** в **`pauk/subagents/`** плюс **`WORKFLOW.md`**. Политика: **`pauk/policy/A-INVARIANTS.md`**. Пресет **`full-pipeline`** (команда **`flow-full-pipeline`**) — **`ARTIFACT-ROUTER.md`** на фазах аналитики и приёмки. Пресет **`technical-design`** (команда **`flow-technical-design`**) — **`ARTIFACT-ROUTER.md`** на первом заходе **`task-analyst`**, **`TECHNICAL-DESIGN-SCAFFOLD.md`** на финальном заходе **`task-analyst`**.

**Навыки из `route.skills`:** для каждого `skill_id` читать `**pauk/skills/<skill_id>/SKILL.md`** (см. раздел «Где лежит тело навыка» в каталоге). Дальнейшие файлы того же навыка — по внутренним ссылкам из `SKILL.md`, без угадывания путей.

## Каталог навыков

Таблица `skill_id` и одной строки описания — в `SKILLS-CATALOG.md` в этой папке.
