# Точка входа и шаг 1 маршрутизации

## Вход задачи


| Поле     | Обязательность |
| -------- | ---------------- |
| Задача   | да               |
| Не делаем | нет             |
| Среда (без секретов) | желательно |
| Явный сигнал **крупной** задачи | нет; см. ниже |

### Масштаб без вопроса пользователю

**По умолчанию** задача считается **локальной** → в `route` ставится `scale: trivial`. **Не задавать** отдельный вопрос «мелкая или нетривиально», если человек сам не просит помочь оценить объём.

**`scale: full`** — только если пользователь **явно** обозначил крупный объём или полный цикл работы. Достаточные примеры формулировок: объёмная / большая задача, нетривиально, полный цикл, с нуля, нужен анализ/дизайн/blueprint на всю фичу, затрагиваем много модулей / интеграция / миграция данных и т.п. Отсутствие таких сигналов → **`trivial`**, даже если задача на самом деле окажется сложной (уточнения — в ходе работы).

Явные слова «мелкая правка» / «мелко» → как правило `scale: trivial`; «нетривиально» в сообщении пользователя → как правило `scale: full`.

## Шаг 1 — маршрут (`route`) для исполнителя

Перед выбором навыков использовать **только** таблицу `skill_id` из `SKILLS-CATALOG.md` в этой папке (при необходимости прочитать файл); итоговый набор `skills` — только перечисленные там идентификаторы, без выдуманных имён.

Исполнитель **обязан** явно для себя зафиксировать маршрут в структуре **`route`** (поля ниже), **как если** он записан в YAML — даже если в ответ пользователю YAML **не** попадает.

Структура (внутренний контракт):

```yaml
route:
  scale: trivial | full
  workflow: []
  skills: []
  blockers: []
```

Соответствие: `scale: trivial` → короткий `workflow` из `pauk/subagents/WORKFLOW.md`; `scale: full` → полная цепочка оттуда же.

### Видимость для пользователя

**По умолчанию** человеку **не** показывать блок YAML `route`: это служебная разметка для дисциплины исполнителя, а не содержательный ответ по задаче 1С.

Допустимо **одно короткое** нейтральное предложение человеческим языком (без обязательного перечисления `skill_id`), например: «Опираюсь на типовые правила по формам и сразу разберу вопрос.» Или **вообще без** вступления — сразу полезный текст по задаче.

**Вывести YAML `route` в чат** — если пользователь **явно** попросил (маршрут, route, отладка оснастки, «покажи skills») или после **словесной** просьбы поправить навыки/цепочку — по желанию одним блоком для ясности.

### Поведение после фиксации маршрута

**`scale: trivial`:** в **том же** сообщении выполнить **шаг 2** — не ждать следующего сообщения пользователя. Прочитать `SKILL.md` по каждому id из `skills`, дальше работать по задаче и `workflow`. Полный формальный артефакт `task-analyst` не обязателен; главное — не обрывать ответ на «пустом» маршруте без ответа по задаче.

**`scale: full`:** в этом сообщении для человека — **короткий** осмысленный текст (1–2 предложения): что будет дальше и что написать в ответ („продолжай“ или уточнение). **Без** развёрнутой реализации и без подключения тел навыков в этом же сообщении. YAML `route` в чат — только по правилам видимости выше.

### Переход к шагу 2 при `scale: full`

**Не требуется** отдельное подтверждение или выбор навыков человеком: набор `route.skills` уже задан на шаге 1.

«Продолжение» — **следующее сообщение пользователя** после ответа с короткой подсказкой (например «продолжай», уточнение по задаче, правка границ scope). Человек **может** попросить изменить навыки / цепочку / масштаб словами — скорректировать внутренний `route` и работать по обновлённому маршруту. Два шага для **full** нужны, чтобы не смешивать в одном сообщении лёгкое планирование и тяжёлый аналитический слой.

## Шаг 2 и далее

**Если шаг 2 уже выполнен в том же сообщении** (`scale: trivial`) — дальше обычный диалог по задаче, политика `pauk/policy/A-INVARIANTS.md`.

**Если был `scale: full`:** после следующего сообщения пользователя — при необходимости снова открыть `pauk/routing/SKILLS-CATALOG.md`, роли субагентов по одной. Политика готовности, запретов и принятия blueprint: `pauk/policy/A-INVARIANTS.md`.

**Навыки из `route.skills`:** для каждого `skill_id` читать `**pauk/skills/<skill_id>/SKILL.md`** (см. раздел «Где лежит тело навыка» в каталоге). Дальнейшие файлы того же навыка — по внутренним ссылкам из `SKILL.md`, без угадывания путей.

## Каталог навыков

Таблица `skill_id` и одной строки описания — в `SKILLS-CATALOG.md` в этой папке.
