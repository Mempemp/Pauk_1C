# Точка входа и шаг 1 маршрутизации

## Вход задачи


| Поле                                        | Обязательность                                                                                |
| ------------------------------------------- | --------------------------------------------------------------------------------------------- |
| Задача                                      | да                                                                                            |
| Не делаем                                   | нет                                                                                           |
| Среда (без секретов)                        | желательно                                                                                    |
| Масштаб: `мелкая_правка` или `нетривиально` | да; если не указан — модель задаёт один короткий вопрос и ждёт ответа, не классифицирует сама |


## Шаг 1 — только `route`

Перед заполнением `route.skills` использовать **только** таблицу `skill_id` из `SKILLS-CATALOG.md` в этой папке (при необходимости прочитать файл); в `skills` — только перечисленные там идентификаторы, без выдуманных имён.

После известного масштаба первый содержательный ответ — только блок:

```yaml
route:
  scale: trivial | full
  workflow: []
  skills: []
  blockers: []
```

Без полного артефакта аналитика, без тел навыков, без длинного плана в том же сообщении.

Соответствие: `мелкая_правка` → обычно `scale: trivial` и короткий `workflow`; `нетривиально` → `scale: full` и цепочка из `pauk/subagents/WORKFLOW.md`.

### Переход к шагу 2 (что значит «продолжение диалога»)

**Не требуется** отдельное подтверждение или выбор навыков человеком: список `route.skills` уже задаёт модель на шаге 1.

«Явное продолжение» — это **следующее сообщение пользователя** после ответа с блоком `route` (например «продолжай», уточнение по задаче, правка границ scope). Человек **может** попросить изменить `route` / `skills` / `workflow` — тогда скорректировать маршрут и дальше работать по обновлённому `route`. Цель двух шагов — не смешивать в одном ответе модели только маршрутизацию и развёрнутую работу (аналитика, тела навыков, длинный план).

## Шаг 2 и далее

После следующего сообщения пользователя после шага 1: при необходимости снова открыть `pauk/routing/SKILLS-CATALOG.md`, роли субагентов по одной. Политика готовности, запретов и принятия blueprint: `pauk/policy/A-INVARIANTS.md`.

**Навыки из `route.skills`:** для каждого `skill_id` читать `**pauk/skills/<skill_id>/SKILL.md`** (см. раздел «Где лежит тело навыка» в каталоге). Дальнейшие файлы того же навыка — по внутренним ссылкам из `SKILL.md`, без угадывания путей.

## Каталог навыков

Таблица `skill_id` и одной строки описания — в `SKILLS-CATALOG.md` в этой папке.
