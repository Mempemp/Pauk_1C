# Каталог навыков

Оценка **`scale`**, поля **`route.workflow`** и **`route.skills`** (классификация и маршрут по Pauk) **не** задаются отдельным `skill_id` в этой таблице: правила в **`pauk/routing/PROTOCOL.md`** и точка входа **`.cursor/rules/pauk-entry.mdc`**.

Поле `route.skills` — **только** идентификаторы из таблицы ниже. Подбор **`skills`** на шаге 1 маршрутизации делается **вместе** с цепочкой **`route.workflow`** (`pauk/subagents/WORKFLOW.md`): активная роль субагента **не заменяет** навыки, а работает **поверх** уже выбранных `SKILL.md`. Когда подмешивать тела навыков — по **`pauk/routing/PROTOCOL.md`**: при `scale: trivial` — в том же первом содержательном сообщении; при `scale: full` — со следующего сообщения пользователя. YAML `route` в чат пользователю по умолчанию не выводится.

## Где лежит тело навыка

Для каждого `skill_id` из таблицы ниже:

- каталог: `pauk/skills/<skill_id>/` (подставить значение из первого столбца);
- **всегда открывать сначала** `pauk/skills/<skill_id>/SKILL.md` — это вход: приоритеты, индекс, ссылки на остальные файлы того же навыка;
- прочие файлы (`templates.md`, `reference/*.md` и т.д.) — только если нужны по задаче, по ссылкам из `SKILL.md`.

| skill_id | Одна строка |
|----------|-------------|
| coding-standards | Стандарты BSL по ИТС: области модулей, директивы клиент/сервер, строки и НСтр, формы, запросы, исключения; при написании, правке и ревью кода 1С |
| error_handling | Транзакции вручную, блокировки данных и объектов, исключения и журнал, пакетная обработка, внешние вызовы; при записи в БД и явном НачатьТранзакцию |
| form_patterns | Модуль управляемой формы и Form.xml: клиент—сервер, один вызов, реквизит↔значение, области модуля, типовые формы, события в XML, жизненный цикл, динамические списки, оформление, диалоги, ТЧ, оповещения, типы, конвейер XML/метаданных |
| ssl_patterns | Паттерны БСП: ОбщегоНазначения, строки, КлиентСервер, журнал, файлы, печать, права; не дублировать библиотеку; при конфигурациях на базе БСП |
| query-patterns | Запросы 1С: без N+1 к СУБД, ВТ и индексы, параметры внутри виртуальных таблиц регистров, ТаблицаЗначений для пакета, NULL и LEFT JOIN, ВЫРАЗИТЬ; при Запрос, остатках/оборотах/срезе, отчётах и массовых выборках |
| metadata_manage | XML-выгрузка конфигурации: объекты метаданных, краткий JSON-DSL, валидация объекта и Form.xml, MCP XSD/verify; без CF/CFE/БД и без поставляемых CLI-утилит. Вместе с form_patterns при правках Form.xml + структуры объекта |
| integration_patterns | Интеграция 1С с внешними системами: сначала прототип/контракт вне клиента 1С (часто Python), затем BSL по отлаженной схеме; HTTP, обмен; вместе с error_handling по внешним вызовам |
| refactor_workflow | Крупный рефакторинг BSL/метаданных: top-down анализ (цепочки, данные) → bottom-up вспомогательные модули → интеграция наверх; остатки имён/контрактов через MCP; см. pauk/mcp/README |
| powershell_windows | Windows PowerShell: цепочки команд, кавычки путей, HTTP/Docker/пауза/JSON в обвязке; не путать с bash; при сценариях вокруг 1cv8, CI, .ps1 на Windows |

## Сигналы → навыки (эвристики для шага 1)

Использовать **только** `skill_id` из таблицы выше. Раздел не вводит новых имён: это подсказки, **какие узкие навыки** почти всегда стоит добавить к **`coding-standards`**, если сигнал есть в **тексте задачи**, **путях к файлам**, **вложениях**, **`@`-ссылках** или в **контексте открытых файлов IDE** (если он передан в сессию). Если сигналов нет — достаточно `coding-standards` и явного смысла задачи.

| Сигнал | Часто уместные skill_id (в дополнение к coding-standards) |
|--------|-------------------------------------------------------------|
| Путь к файлу заканчивается на **`Form.Module.bsl`**, или содержит **`Формы`** / **`Forms`** и фрагмент вроде **`...\Форма\Ext\Form\Module.bsl`** / **`...Form\Module.bsl`** (модуль управляемой формы в выгрузке или EDT); либо явно **`Form.xml`** формы | **form_patterns** |
| В задаче явно: модуль формы, обработчик элемента формы, перенос данных клиент↔сервер **с формы**, динамический список на форме, ТЧ формы, оповещение между формами | **form_patterns** |
| Текст или правка **`Запрос`**, виртуальные таблицы регистров, остатки/обороты/срез, СКД, массовая выборка, оптимизация запроса | **query-patterns** |
| БСП, вызовы **`ОбщегоНазначения`** и других типовых модулей библиотеки, сценарий «как в типовой» | **ssl_patterns** |
| **`НачатьТранзакцию`**, блокировки данных/объектов, запись в БД с контуром ошибок, пакет с откатом, внешние вызовы с обработкой сбоев | **error_handling** |
| Путь к **`Configuration.xml`**, каталогам выгрузки **`Catalogs/`**, **`Documents/`**, **`InformationRegisters/`** и т.п.; явно: **объект метаданных**, **реквизит/ТЧ в XML**, **валидация метаданных**; правка **`Form.xml`** как структуры (не только модуль) | **metadata_manage** (+ часто **form_patterns** для модуля и поведения) |
| **Интеграция** с внешней системой, **HTTP**/REST, **обмен**, **очереди**, **файловый/сетевой** контракт, явно **Python**-прототип до BSL, **клиент API** | **integration_patterns** (+ обычно **error_handling** при таймаутах/записи) |
| **Рефакторинг**, **переименование** сущностей/методов, **вынесение** в общий модуль, **связность** и **влияние** изменений по конфигурации, разбор **call graph** / **usages** | **refactor_workflow** (часто вместе с **query-patterns** / **form_patterns** / **metadata_manage** — по смыслу) |
| Задача на **PowerShell** / **.ps1**, **Windows**-терминал, цепочка вроде `cd` + команда, **docker-compose** на Windows, **Invoke-WebRequest**, правка/отладка **скрипта** выгрузки (не синтаксис BSL) | **powershell_windows** |

Несколько сигналов — **несколько** узких навыков в `route.skills`; один и тот же `skill_id` не повторять.
