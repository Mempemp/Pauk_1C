# Бэклог фабрики (вне `pauk-product/`)

Идеи и долг, которые **не** входят в копируемый пакет. Реализация по приоритету.

| ID | Тема | Суть | Статус |
|----|------|------|--------|
| BL-001 | MCP для БСП | Версий БСП и объёма кода в типовых конфигурациях много: постоянный `grep` по выгрузке тяжёл и не гарантирует соответствие **вашей** ревизии. Нужен **отдельный слой** быстрого поиска: индексация по **конкретной** поставке/версии БСП, семантический или символьный поиск, контракт «какой снимок/версия проиндексирована». Рассматривать **после** доработок по поиску в репозитории и (при необходимости) универсальных code-MCP (Serena, RLM и т.п.) — уточнить, не закрывают ли они сценарий. | open |
| BL-002 | Субагент и навыки (skills) | Субагенту нужно уметь **подбирать релевантные навыки** из доступного набора. Без предварительного **разбиения задачи на составляющие** и **классификации** (тип работы, домен, инструменты) корректный выбор навыка практически невозможен: иначе совпадение по ключевым словам или эвристике будет хуже человеческого «понимания контекста». Продумать пайплайн: декомпозиция → теги/классы → маппинг на `SKILL.md` / метаданные навыков. **Возможно** зафиксировать те же требования в **правиле** (оркестратор / субагент): явная фаза классификации до вызова навыков. | open |
| BL-003 | Правила Comol → наша система | Разобрать и **сконвертировать** набор [comol/cursor_rules_1c](https://github.com/comol/cursor_rules_1c) под структуру фабрики (правила, субагенты, навыки, оркестрация). В апстриме уже много готового — **селективно переносить** и адаптировать, а не дублировать с нуля. Локальная копия для анализа: `Материалы к конвертации/cursor_rules_1c-main/`. | open |
| BL-004 | Менеджмент контекста + поиск по коду | Снизить расход токенов за счёт **целенаправленной подачи контекста**: как минимум **MCP для поиска по кодовой базе** (символьный/семантический — уточнить), чтобы агент запрашивал релевантные фрагменты вместо широкого `read_file` / массового grep. Согласовать с BL-001: отдельный индекс для БСП vs. единый слой по репозиторию продукта. | open |
| BL-005 | Английский для продуктовой оснастки | Перевести на **английский** правила, навыки, инструкции субагентов и прочую **продуктовую** обвязку, которая постоянно попадает в контекст, чтобы **сократить расход токенов** (английский обычно компактнее и лучше совпадает с токенизаторами/кодовой базой). Сохранить согласованность терминов; при необходимости — двуязычные глоссарии вынести в отдельные короткие файлы. | open |

## BL-001 — детализация (критерий готовности, черновик)

- Можно по запросу разработчика найти **релевантные** модули/процедуры БСП **в той ветке версии**, с которой работают, без полного прогона репо вручную.
- Понятно, **какая** редакция БСП лежит в индексе; при смене поставки — явное обновление снимка.
- Тело навыка `ssl_patterns` остаётся **без** ссылок на внутреннюю кухню фабрики; про этот инструмент можно описать **в продукте** только как «при подключённом MCP …», когда MCP появится.

## BL-002 — детализация (черновик)

- Есть явный шаг между «сырой» постановкой и выбором навыков: **декомпозиция** и **метки** (например: платформа, CI, документация, оркестрация агентов).
- Критерий подбора навыка привязан к **классификации**, а не только к текстовому совпадению с описанием в `description`.
- Решено, дублируется ли это в **cursor rule** / инструкции оркестратора (один источник правды vs. дублирование).

## BL-003 — детализация (черновик)

- Инвентаризация: что в репозитории относится к **rules**, **skills**, **субагентам**, вспомогательным скриптам/докам — и что из этого совместимо с текущим раскладом `pauk-product/` и фабрики.
- **Правила конвертации**: сопоставить globs, frontmatter, имена; убрать/заменить пути и ссылки, специфичные для оригинала; зафиксировать единый чек-лист «Comol → Pauk».
- **Не слепое копирование**: взять паттерны, формулировки и идеи; при конфликте с уже принятыми правилами фабрики — осознанно выбрать вариант или merge.
- Критерий готовности: принято решение по каждой крупной сущности (взяли / адаптировали / отложили) и обновлёны артефакты фабрики либо задокументировано отложенное.

## BL-004 — детализация (черновик)

- **Цель**: меньше «лестницы» из полных чтений файлов; больше точечных hit’ов с путями и выдержками.
- Минимальный шаг: подключить и настроить MCP поиска по репо (или зафиксировать, почему выбран встроенный инструмент + политика использования).
- Политика для агентов/правил: сначала поиск/индекс, затем раскрытие файла; границы scope (только `pauk-product/`, вся фабрика, исключения).
- Связь с **BL-001**: не плодить два независимых поиска без схемы; приоритет — один контракт «где ищем».

## BL-005 — детализация (черновик)

- **Объём**: то, что копируется в пакет продукта и длительно в контексте (`.cursor/rules`, `SKILL.md` в продукте, промпты субагентов, пользовательские инструкции пакета).
- **Исключения/осторожность**: внутренняя документация фабрики, обсуждения с заказчиком, юридические формулировки — не переводить слепо; список исключений.
- **Контроль качества**: единый глоссарий 1С-терминов (EN) или ссылка на внешний; вычитка носителем по критичным инструкциям.
- Критерий готовности: основной продуктовый пакет на EN; в README/релиз-ноутах указано, что EN — рабочий язык инструкций.
