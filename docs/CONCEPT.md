# Концепция: слои, навыки, маршрутизация, субагенты

Документ для обсуждения. Решения помечайте явно (например, блок «Принято» в конце раздела).

---

## 1. Цель оснастки

- Высокое качество и автономность выполнения задач по разработке на платформе 1С и смежным доменам.
- **Универсальность относительно модели**: смена модели/провайдера/локального рантайма не ломает «корневой» слой.
- Расширяемость: в перспективе MCP (свои и опенсорс), без проектирования ядра «под один MCP».

---

## 2. Предлагаемые архитектурные слои

Ниже — логическое разделение, не обязательно отдельные репозитории.

| Слой | Содержание | Зависимости |
|------|------------|-------------|
| **A. Политика и контракт задачи** | Что считается «готово», формат ввода/вывода, чеклисты, безопасность (секреты, прод) | Ни от кого — см. [LAYER-A.md](LAYER-A.md) |
| **B. Оркестрация** | Разбиение на подзадачи, вызов субагентов, merge результатов, остановка по критериям | A |
| **C. Доменные навыки (Skills)** | 1С BSL, метаданные, запросы, интеграции, конкретные подсистемы конфигурации | A; при необходимости B |
| **D. Маршрутизация навыков** | Как из описания задачи выбрать 1–N навыков (или «общий» fallback) | A, каталог C |
| **E. Инструменты** | MCP, CLI, RAG, поиск по базе знаний | Контракты из A/D |

**Инвариант:** слои A–D не ссылаются на конкретную LLM API. Исполнитель — «черный ящик», которому передаются текст инструкций + (опционально) инструменты.

### Репозиторий: фабрика и продукт

Этот Git-репозиторий — **фабрика** (`docs/` и мета-разработка). В прикладной репозиторий копируется только содержимое пакета **`pauk-product/`**. Редакционные правила пакета (никакого мета-текста для модели в проде) — **`docs/PRODUCTION-BUNDLE.md`**.

### 2.1. Семантический слой задачи (смысл ↔ навыки оси 1)

Отдельно от **оси 2 навыков** (узкий прикладной домен конкретной конфигурации) полезно явно держать **семантический слой**: из формулировки задачи вывести *что именно* нужно сделать в терминах рисков, последовательностей и недостающих данных — и **перечень релевантных навыков оси 1** (платформа: запросы, формы, клиент-сервер, метаданные, HTTP, конвертация и т.д.).

Этот слой по смыслу ближе всего к **оркестрации (B)** и артефакту планирования: не дублирует ИТС, а задаёт **логику работы** и карту «какие компетенции подключать дальше».

**Пример (обмен конфигурациями):** недостаточно записать данные в базу; часто нужна постобработка, вплоть до **имитации пользовательского заполнения** (скрытые реквизиты, типовая логика заполнения документа). Данных из источника не хватает — нужно **исследовать объект** и понять, как он «дозаполняется» в пользовательском режиме. Это трудоёмко в реализации, но **концептуально** это: признание класса проблем + шаги анализа + связка с навыками (метаданные, модули объекта/формы, обработчики, клиент-сервер, возможно конвертация).

**Куда класть такое знание:**

| Вариант | Когда уместно |
|---------|----------------|
| **Субагент / playbook оркестрации** | Методология **класса задач** (обмен, миграция, «данные есть, семантики нет»): чеклист вопросов, типовые ловушки, порядок исследования, **какие навыки оси 1** типично нужны — без привязки к одной конфигурации. Результат: явный план и список skill id для следующих шагов. |
| **Ось 2 (навык)** | Знание **стабильно повторяется** для линейки продуктов или конкретной пары конфигураций (соглашения, известные точки расширения, типовой состав постобработки *в вашем контексте*). |
| **Навык оси 1** | Только если формулировка **универсальна для платформы** (например, как в принципе вызывается заполнение, типовые механизмы платформы) — без сценария «наш обмен всегда так». |

**Принято для обсуждения:** концептуальная логика «обмен ≠ просто INSERT» и схема мышления **не обязана** жить в оси 2; по умолчанию она — **в оркестрации / роли субагента** (или в отдельном лёгком playbook-документе, который читает этот субагент). Ось 2 подключается, когда появляется **повторяемый прикладной пакет** под ваши конфигурации.

## 3. Навыки (Skills): гранулярность и жизненный цикл

### Оси (согласовано в обсуждении)

| Ось | Содержание | Решение на сейчас |
|-----|------------|-------------------|
| **1 — платформа / технологии** | Запросы, модули, формы, клиент-сервер, HTTP/API, метаданные, конвертация данных, логирование и т.п. | **Основной фокус** первых навыков; универсально для любых конфигураций. |
| **2 — прикладной домен** | Подсистема или конкретная конфигурация | **Не разворачиваем на старте**: процессы между конфигурациями расходятся; возможны узкие навыки «под объект» позже. Задел — в каталоге/метаданных, без обязательства. |
| **3 — процессы** | Ревью, пайплайн задач, роли | **В субагентах**; в навыках максимум — **конфигурация workflow** (когда какой шаг и какие артефакты передаются), без замены роли исполнителя. |

Вопросы ещё открыты:

- Версионирование: semver в описании навыка, changelog?
- Пересечения: два навыка противоречат друг другу — приоритет правил редактора vs навыка vs системного промпта?

**Стартовая линия:** осмысленные узкие навыки по оси 1; содержание выводить из документации 1С и проверенных практик, **сжимая суть** в один или несколько связанных навыков при большом объёме материала. Цель по смыслу — аналог «головы Senior»: выборочная подгрузка знаний под задачу (плюс маршрутизация отдельно).

### Как собираем навык (принято на уровне принципа)

Навык — это **не** копипаста ИТС в промпт. Зафиксированный порядок:

1. **Источники** — официальная документация платформы (и при необходимости типовые методики / устойчивые практики с явным указанием «откуда взято» и версии платформы, если релевантно).
2. **Отбор** — что реально нужно для принятия решений в коде (критерии, ограничения, типовые ошибки, предпочтительные приёмы), без «воды».
3. **Сжатие** — формулировки для модели: короткие правила, чеклисты, граничные случаи; при перегрузе — **разбиение на несколько связанных навыков** или вынесение справочного болота в MCP/RAG с оставлением в навыке только ссылки на контракт использования. При разбиении **тела одного навыка** на файлы под `pauk-product/pauk/skills/<skill_id>/` — **нарезка по логическим смыслам** и **тематические имена** в `reference/` (не по номерам правил из источника); операционная спецификация структуры и именования — в правиле фабрики `.cursor/rules/pauk-factory.mdc` (раздел «Навыки в pauk-product»).
4. **Метаданные для каталога** — id, описание, триггеры для маршрутизации (добавляются при публикации навыка в оснастку, не обязательно в том же файле, что тело).

Итог: **сначала опора на документацию и проверяемые практики, затем осмысленная подготовка тела навыка**; обратный порядок («сначала текст навыка, потом подтянуть цитаты») не является целевым.

### «Стандарты 1С / типовые ошибки» — навык, правило или субагент? (на обсуждении)

| Вариант | Плюсы | Минусы |
|---------|--------|--------|
| **Правило** | Всегда активно; сильные инварианты (безопасность, запреты). | Раздувает каждый запрос; плохо масштабируется, если туда тащить весь ИТС. |
| **Навык** | Подключается по релевантности; можно версионировать и уточнять без смены «глобального» тона. | Риск дублирования с узкими навыками, если не держать границу «общие принципы vs детали домена». |
| **Субагент** | Удобно для **проверяющей** роли (чеклист, ревью). | Не заменяет справочные знания при **написании** кода в незнакомом домене — исполнителю всё равно нужен контекст. |

**Предложение зафиксировать как рабочую гипотезу:** жёсткие **инварианты** (что нельзя, прод, секреты) — в **rules**; **универсальные стандарты оформления и типовые антипаттерны платформы**, не привязанные к одному модулю — в **один компактный навык** «платформа: общие стандарты» (или разбить позже, если раздуется); **формальная проверка** по чеклисту — этап **субагента** в workflow. Итог можно оформить ADR после пилота.

### Язык навыков

- **Сейчас:** русский.
- **Заранее:** при росте контекста — перевод тел навыков на английский для экономии токенов (отдельное решение по мере готовности каталога).

## 4. Маршрутизация навыков

Варианты (можно комбинировать):

1. **Статический реестр** — таблица «ключевые слова / теги → skill id» в репозитории; просто, прозрачно, плохо масштабируется при сотнях навыков.
2. **Двухшаговый LLM-вызов** — сначала классификация задачи → список skill id; затем основная работа с приложенными навыками. Зависит от модели для маршрутизации, но **логика маршрутизации** остаётся отдельным артефактом (промпт + схема ответа).
3. **Внешний MCP «skill-registry»** — хранит метаданные навыков (описание, триггеры, embeddings); агент спрашивает MCP «что релевантно?». Удобно для большого числа навыков и обновлений без правки `.cursorrules`.
4. **Embeddings / RAG локально** — то же, что 3, но без MCP, через локальный индекс.

**Принято в v1 (Pauk, BL-002):** в продуктовом пакете **статический** реестр (п.1) в [`SKILLS-CATALOG.md`](../pauk-product/pauk/routing/SKILLS-CATALOG.md) дополняется **явной** фазой для объёмных задач: handoff-шаблон [`ARTIFACT-ROUTER.md`](../pauk-product/pauk/routing/ARTIFACT-ROUTER.md), нулевой этап ролью `task-analyst` и **предопределённый** сценарий `full-pipeline` (см. [`flow-full-pipeline.md`](../pauk-product/.cursor/commands/flow-full-pipeline.md) и граф в [`WORKFLOW.md`](../pauk-product/pauk/subagents/WORKFLOW.md)). Обычный режим — без отдельного файла-команды. П.2 (двухшаговый сценарий) частично совпадает: классификация → `route` → дальше работа с `SKILL.md` по `route.skills`. «Истина» о списке `skill_id` — по-прежнему **в git** (таблица каталога).

**Открытый вопрос (масштабирование):** дублирование метаданных навыков в **сервисе** индексации (п.3–4) — при существенном росте каталога.

## 5. Субагенты и workflow

Полезные паттерны:

- **Explore → Plan → Implement → Verify** (обзор кода → план → правки → проверка линтеров/тестов).
- **Специализированные роли:** например, «только чтение и поиск», «только правки BSL», «только документация» — снижает риск побочных diff.
- **Явный handoff:** что передаётся следующему субагенту (артефакт: файл решений, список файлов, ограничения).

### Крупноблочный псевдокод и архитектура реализации (зафиксировано)

При проектировании **крупного** объёма кода сначала нужен артефакт **крупноблочного псевдокода**: зависимости между блоками, порядок и цепочки вызовов, границы ответственности модулей/процедур — без ухода в детальную реализацию. Это влияет на качество итогового кода.

**Обязательность:** не глобальное правило для каждой мелкой правки. Логика **условного** включения шага зашивается в **оркестрацию / субагента** (оценка масштаба задачи: маленькая точечная правка → шаг пропускается; новая нетривиальная фича / много модулей / неочевидные зависимости → шаг выполняется). «Махина» не вызывается без смысла.

**Где размещать логику этого шага:**

- **Субагент (предпочтительно):** отдельная роль вроде «Design / Blueprint» с явным выходом (структурированный документ или схема), **без** правок кода до принятия плана. Совместимо с универсальностью от модели: инструкция роли — текст, формат выхода — контракт слоя A.
- **Навык:** краткие **критерии хорошего** псевдокода/шаблон разделов (что обязательно показать: вызовы, данные, ошибки, границы сервер/клиент). Не дублировать полный workflow — только знание «как оформлять проектирование».
- **Правило:** не требуется как «всегда псевдокод»; при необходимости — только инварианты слоя A без привязки к размеру задачи.

Итог: **условный шаг в workflow субагентов** + при необходимости **навык** с критериями артефакта; критерий «когда включать» — в описании оркестрации/роли, а не в глобальных rules.

Черновик зафиксирован в [SUBAGENTS.md](SUBAGENTS.md): роли, условия, артефакты, handoff.

## 6. Правила (Rules) редактора

Правила обычно **глобальные и жёсткие** (стиль, безопасность, язык ответа). Навыки — **контекстные**. Избегать дублирования: в правилах — инварианты; в навыках — домен.

## 7. MCP в будущем

Слой E подключается через стабильные интерфейсы:

- «Запрос к платформе 1С» (типы, методы) — кандидат в MCP.
- **БСП и тяжёлые справочные контуры** — кандидат вынести в MCP/RAG (актуальность, объём, ссылки на релиз), а в навыках оставить только **сжатые** паттерны взаимодействия с подсистемами, если они стабильны для вашего процесса.

Ядро (A–D) описывает **что** нужно от инструмента, не **как** он реализован.
