# Точка входа и маршрутизация

**Проблема:** полные `CONCEPT`, `SUBAGENTS`, `LAYER-A` — **источник правды для людей и сборки оснастки**, а не текст по умолчанию в каждый запрос к модели.

**Принцип:** на вход — **минимум**; после первого шага маршрутизации подмешиваются только выбранные фрагменты (тела навыков, одна роль субагента и т.д.).

Дизайн маршрутизации для фабрики. Рабочий контракт без лишнего контекста — в пакете `pauk-product/pauk/routing/PROTOCOL.md` (оформление пакета: `docs/PRODUCTION-BUNDLE.md`).

---

## 1. Точка входа (шаблон для человека)

Один стабильный формат сообщения:

| Поле | Содержание |
|------|------------|
| Задача | Что сделать, в границах одной итерации. |
| Не делаем | Опционально: явные исключения по scope. |
| Среда | База / релиз платформы / ветка — **без** секретов и строк подключения. |
| **Крупная задача** | Опционально: явные слова в духе объёмная / большая / нетривиально / полный цикл / с нуля / много модулей. Без такого сигнала действует **умолчание «обычная задача»** (`scale: trivial` во внутреннем маршруте). |

Модель **не** задаёт отдельный вопрос «мелкая или нетривиально», пока человек сам не просит помочь оценить объём.

Дополнительно по желанию: известные файлы / объекты метаданных.

Объём: **порядка одного экрана**, без README и без «прочитай весь `docs/`».

## 2. Шаг 1 — маршрут и что дальше

Исполнитель **перед** ответом опирается на сжатый реестр `skill_id` в **`pauk-product/pauk/routing/SKILLS-CATALOG.md`** (в целевом репозитории после копирования пакета — `pauk/routing/SKILLS-CATALOG.md`). В набор `skills` внутреннего маршрута `route` — **только** идентификаторы из таблицы каталога, без выдуманных имён.

Структура `route` (поля `scale`, `workflow`, `skills`, `blockers`) — **контракт для исполнителя**; в чат пользователю YAML по умолчанию **не** выводится (см. [PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md), раздел про видимость).

**`scale: trivial` (по умолчанию):** в **том же** первом содержательном сообщении — работа по задаче с телами навыков по выбранным `skill_id`. Обрыв без ответа по задаче недопустим.

**`scale: full` (только при явном сигнале крупной задачи):** короткий текст с подсказкой (следующее сообщение запускает шаг 2); без тел навыков и без развёрнутой реализации здесь же. Полная цепочка ролей — по [SUBAGENTS.md](SUBAGENTS.md) (с условным `design`).

Критерии `trivial` / `full` — в [pauk-product/pauk/routing/PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md).

---

## 3. Шаг 2 и далее

**Если был `scale: trivial`:** шаг 2 уже совмещён с шагом 1 в одном ответе; дальше обычный диалог.

**Если был `scale: full`:** шаг 2 начинается после **следующего сообщения пользователя** (например «продолжай», уточнение задачи, смена границ scope). **Отдельное подтверждение списка навыков не требуется:** набор навыков уже задан на шаге 1. Человек **может** попросить скорректировать маршрут словами или **попросить показать YAML `route`** — дальше работа по обновлённому маршруту.

- Подмешиваются **тела** навыков только по `skills` из `route`.
- Подключается **одна** роль субагента за раз (например `task-analyst`), по [SUBAGENTS.md](SUBAGENTS.md).
- Инварианты слоя A в промпте — **коротко из rules**, не целиком [LAYER-A.md](LAYER-A.md), пока человек явно не попросит политику.

---

## 4. Сжатый каталог для шага 1

Для заполнения `skills` в первом ответе модель использует только **сжатый реестр** в репозитории оснастки: **`pauk-product/pauk/routing/SKILLS-CATALOG.md`** — таблица `skill_id` + одна строка описания на навык, без полных `SKILL.md`. После копирования пакета в прикладной проект путь к каталогу: `pauk/routing/SKILLS-CATALOG.md`.

---

## 5. Почему два шага только для `full`

Для **обычных** задач (`scale: trivial`) выбор маршрута и работа идут **в одном** ответе — пользователю не нужен промежуточный YAML.

Для **крупных** (`scale: full`) два шага в диалоге сохраняют разделение: сначала короткая подсказка, затем — тяжёлый слой (аналитик, длинные планы, навыки по объёму). Граница — **смена хода**, не голосование по списку навыков.
