# Точка входа и маршрутизация

**Проблема:** полные `CONCEPT`, `SUBAGENTS`, `LAYER-A` — **источник правды для людей и сборки оснастки**, а не текст по умолчанию в каждый запрос к модели.

**Принцип:** на вход — **минимум**; после первого шага маршрутизации подмешиваются только выбранные фрагменты (тела навыков, одна роль субагента и т.д.).

Дизайн маршрутизации для фабрики. Рабочий контракт без лишнего контекста — в пакете `pauk-product/pauk/routing/PROTOCOL.md` (оформление пакета: `docs/PRODUCTION-BUNDLE.md`).

---

## 1. Точка входа (шаблон для человека)

Один стабильный формат сообщения:

| Поле | Содержание |
|------|------------|
| Задача | Что сделать, в границах одной итерации. |
| Не делаем | Опционально: явные исключения по scope. |
| Среда | База / релиз платформы / ветка — **без** секретов и строк подключения. |

Объёмный контур с поэтапным стартом — **не** через ключевые слова в сообщении, а через команду из **`.cursor/commands/`** (пресеты в **`pauk-product/pauk/subagents/WORKFLOW.md`**).

Модель **не** задаёт отдельный вопрос «мелкая или нетривиально», пока человек сам не просит помочь оценить объём.

Дополнительно по желанию: известные файлы / объекты метаданных.

Объём: **порядка одного экрана**, без README и без «прочитай весь `docs/`».

---

## 2. Шаг 1 — маршрут и что дальше

Исполнитель **перед** ответом опирается на сжатый реестр `skill_id` в **`pauk-product/pauk/routing/SKILLS-CATALOG.md`** (в целевом репозитории после копирования пакета — `pauk/routing/SKILLS-CATALOG.md`). В набор `skills` внутреннего маршрута `route` — **только** идентификаторы из таблицы каталога, без выдуманных имён.

Структура `route` (поля `workflow`, `skills`, `blockers`) — **контракт для исполнителя**; в чат пользователю YAML по умолчанию **не** выводится (см. [PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md), раздел про видимость).

**Обычный ввод** (без команды-пресета): в **том же** первом содержательном сообщении — работа по задаче с телами навыков по выбранным `skill_id`. Обрыв без ответа по задаче недопустим.

**Пресет из команды** (`flow-full-pipeline`, `flow-technical-design`): **режим первого ответа** и цепочка ролей — в [WORKFLOW.md](../pauk-product/pauk/subagents/WORKFLOW.md) (таблица пресетов); без дублирования в [PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md).

---

## 3. Шаг 2 и далее

**Обычный ввод:** шаг выполнения уже в первом ответе; дальше обычный диалог.

**Пресет с поэтапным стартом:** продолжение после **следующего** сообщения пользователя — см. [PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md) и [WORKFLOW.md](../pauk-product/pauk/subagents/WORKFLOW.md). **Отдельное подтверждение списка навыков не требуется:** набор навыков уже задан на шаге 1. Человек **может** попросить скорректировать маршрут словами или **попросить показать YAML `route`**.

- Подмешиваются **тела** навыков по правилам пресета и `PROTOCOL.md`.
- Подключается **одна** роль субагента за раз (например `task-analyst`), по [SUBAGENTS.md](SUBAGENTS.md).
- Инварианты слоя A в промпте — **коротко из rules**, не целиком [LAYER-A.md](LAYER-A.md), пока человек явно не попросит политику.

---

## 4. Сжатый каталог для шага 1

Для заполнения `skills` в первом ответе модель использует только **сжатый реестр** в репозитории оснастки: **`pauk-product/pauk/routing/SKILLS-CATALOG.md`** — таблица `skill_id` + одна строка описания на навык, без полных `SKILL.md`. После копирования пакета в прикладной проект путь к каталогу: `pauk/routing/SKILLS-CATALOG.md`.

---

## 5. Два шага в диалоге только у объёмных пресетов

Для **обычного** ввода выбор маршрута и работа идут **в одном** ответе — пользователю не нужен промежуточный YAML.

Для пресетов **`full-pipeline`** и **`technical-design`** короткий первый ответ и продолжение со следующего сообщения заданы в **`WORKFLOW.md`** (колонка «Режим первого ответа»). Граница — **смена хода**, не голосование по списку навыков.

---

## 6. Пресет объёмного сценария и handoff (классификация, BL-002 v1)

**Обычный** сценарий **отдельного файла-команды не требует** — см. [README.md](README.md) (использование) и [PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md).

Для **заранее заданной длинной цепочки** (пресет `full-pipeline`) в пакете лежит файл **[`pauk-product/.cursor/commands/flow-full-pipeline.md`](../pauk-product/.cursor/commands/flow-full-pipeline.md)**. В **Cursor** его можно вызвать из палитры, если поддерживается папка `.cursor/commands/`; **в иных** средах — открыть тот же markdown вручную. Он ссылается на **«Предопределённые сценарии (пресеты)»** в [`pauk-product/pauk/subagents/WORKFLOW.md`](../pauk-product/pauk/subagents/WORKFLOW.md) и на шаблон **[`pauk-product/pauk/routing/ARTIFACT-ROUTER.md`](../pauk-product/pauk/routing/ARTIFACT-ROUTER.md)**.

`pauk-entry` в пакете содержит шаг **0**: при **явном** задействовании подходящего файла из `.cursor/commands/` исполнитель выравнивается с пресетом до применения остальных пунктов правила. Подробности — в [PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md) пакета.

Связь с бэклогом: [BL-002](BACKLOG.md) (секция «BL-002 — детализация») закрыт на уровне **v1** — строка в таблице бэклога.
