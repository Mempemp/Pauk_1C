# Точка входа и маршрутизация

**Проблема:** полные `CONCEPT`, `SUBAGENTS`, `LAYER-A` — **источник правды для людей и сборки оснастки**, а не текст по умолчанию в каждый запрос к модели.

**Принцип:** на вход — **минимум**; после первого шага маршрутизации подмешиваются только выбранные фрагменты (тела навыков, одна роль субагента и т.д.).

Дизайн маршрутизации для фабрики. Рабочий контракт без лишнего контекста — в пакете `pauk-product/pauk/routing/PROTOCOL.md` (оформление пакета: `docs/PRODUCTION-BUNDLE.md`).

---

## 1. Точка входа (шаблон для человека)

Один стабильный формат сообщения:

| Поле | Содержание |
|------|------------|
| Задача | Что сделать, в границах одной итерации. |
| Не делаем | Опционально: явные исключения по scope. |
| Среда | База / релиз платформы / ветка — **без** секретов и строк подключения. |
| **Масштаб** | Обязательно одно из: **`мелкая_правка`** \| **`нетривиально`**. Задаёт **человек**. |

Если человек **не указал масштаб**, модель **не** строит полный `route` и **не** переходит к шагу 2 сама: она **задаёт вопрос** и предлагает выбрать `мелкая_правка` или `нетривиально` (кратко, без воды).

Дополнительно по желанию: известные файлы / объекты метаданных.

Объём: **порядка одного экрана**, без README и без «прочитай весь `docs/`».

---

## 2. Шаг 1 — только маршрутизация (принято: жёсткое разделение)

После того как **масштаб известен** (из сообщения или из ответа человека на вопрос), исполнитель **перед** ответом опирается на сжатый реестр `skill_id` в **`pauk-product/pauk/routing/SKILLS-CATALOG.md`** (в целевом репозитории после копирования пакета — `pauk/routing/SKILLS-CATALOG.md`). В поле `skills` блока `route` — **только** идентификаторы из таблицы каталога, без выдуманных имён.

**Первый содержательный ответ модели** — **исключительно** короткий блок маршрутизации, без развёрнутого аналитического артефакта и без тел навыков:

```text
route:
  scale: trivial | full
  workflow: [...]
  skills: [skill-id, ...]
  blockers: [...]
```

Соответствие: `мелкая_правка` → как правило `scale: trivial` и укороченный `workflow`; `нетривиально` → `scale: full` и полная цепочка по [SUBAGENTS.md](SUBAGENTS.md) (с условным `design`).

В этом же ответе **запрещено**: полный артефакт `task-analyst`, длинные планы реализации, вставка содержимого навыков. Полная механика формата и перехода к шагу 2 — в [pauk-product/pauk/routing/PROTOCOL.md](../pauk-product/pauk/routing/PROTOCOL.md).

---

## 3. Шаг 2 и далее

Начинается после **следующего сообщения пользователя** после ответа модели, в котором был только блок `route` (например «продолжай», уточнение задачи, смена границ scope). **Отдельное подтверждение или выбор навыков человеком не требуется:** список `route.skills` уже сформирован на шаге 1. Человек **может** попросить скорректировать `route` / `skills` / `workflow` — дальше работа по обновлённому маршруту.

- Подмешиваются **тела** навыков только по `skills` из `route`.
- Подключается **одна** роль субагента за раз (например `task-analyst`), по [SUBAGENTS.md](SUBAGENTS.md).
- Инварианты слоя A в промпте — **коротко из rules**, не целиком [LAYER-A.md](LAYER-A.md), пока человек явно не попросит политику.

---

## 4. Сжатый каталог для шага 1

Для заполнения `skills` в первом ответе модель использует только **сжатый реестр** в репозитории оснастки: **`pauk-product/pauk/routing/SKILLS-CATALOG.md`** — таблица `skill_id` + одна строка описания на навык, без полных `SKILL.md`. После копирования пакета в прикладной проект путь к каталогу: `pauk/routing/SKILLS-CATALOG.md`.

---

## 5. Почему два шага

Предсказуемость оснастки: один тип ответа на шаге 1, один набор последующих действий на шаге 2. Смешивание маршрутизации и `task-analyst` в одном сообщении **не** используем.

**Терминология (согласовано с продуктовым пакетом):** граница шагов — это **смена хода в диалоге**, а не обязательный «человеческий gate» на список навыков. Формулировка «явное продолжение» в ранних обсуждениях означала именно **следующее сообщение пользователя** после шага 1, без требования отдельно утвердить `skills`.
